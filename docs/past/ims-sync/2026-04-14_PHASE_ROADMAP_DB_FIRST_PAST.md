# 2026-04-14 18:42 / f5555a8 / PHASE ROADMAP - DB FIRST

## 목표
**현재 구현되어 있는 홈페이지 구조를 우리 DB를 통해 직접 구현한다.**

즉, 현재의
- 메인 검색
- 차량 목록
- 차량 상세
- 예약 준비
흐름은 유지하되,
그 내부 데이터 소스를 점진적으로
**partner 상세 의존 구조 → 우리 DB + 우리 서버 계산 구조**로 전환한다.

---

## 이번 기준에서 잠그는 핵심 결론

### 1. 차량 DB는 이미 있다고 본다
- 차량 기본 정보 source of truth 는 우리 DB다.
- 차량명, 이미지, 연식, 연료, 좌석, 옵션, 연령 조건 등은 우리 DB 기준으로 간다.

### 2. 예약 DB는 별도 운영이 맞다
- 예약 데이터는 **IMS 예약 정보를 준실시간으로 반영하는 별도 예약 DB/테이블**로 가는 것이 타당하다.
- 이유:
  - 검색 시 가용성 판단은 partner HTML 파싱보다 예약 상태 기반이 더 안정적이다.
  - 상세/결제/확정가 검증은 예약 데이터와 붙어야 한다.
  - 이후 중복 예약 차단, 변경 이력, 취소, 동기화 실패 복구까지 생각하면 별도 예약 테이블이 필요하다.

### 3. 검색은 페이지 입력값 기준으로 시작한다
- 검색 입력은 프론트가 받는다.
- 서버는 그 검색 조건으로
  1) 차량 후보를 찾고
  2) 예약 DB로 가용성을 검증하고
  3) 가격 테이블로 금액을 계산하고
  4) 결과 snapshot을 저장한 뒤
  5) 목록/상세를 렌더한다.

### 4. 기준 흐름은 이것으로 고정한다
**차량 + 예약기간 → 가용성 확인 → 가격 확정 → 출력**

이 흐름이 모든 페이지와 API의 중심이다.

## 목표 아키텍처

### 사용자 흐름
1. 사용자가 메인에서 검색 조건 입력
2. 서버가 조건 기준으로 가용 차량 계산
3. 서버가 차량별 가격 계산
4. 결과를 목록에 출력
5. 목록 snapshot 을 search session 으로 저장
6. 상세는 `searchSessionId + carId` 로만 조회
7. 예약 요청 시 서버가 다시 최종 검증 후 예약 생성

### 서버 흐름
1. 검색 조건 수신
2. 차량 DB 조회
3. 예약 DB 조회로 기간 겹침/가용성 판단
4. 가격 테이블 기준 계산
5. 목록 snapshot 저장
6. 상세 snapshot 제공
7. 예약 확정 시 재검증 + 예약 저장

## 이번 기준에서 필요한 데이터 축

### A. 차량 DB
이미 존재한다고 본다.

역할:
- 차량 기본 정보
- 표시용 기본 메타
- 운영 가능 여부 기본값
- 가격 규칙 연결 키

최소 책임:
- `car_id`
- 표시용 차량 정보
- 연령/면허 조건
- 업체 연결
- 가격 규칙 연결

### B. 가격 테이블
반드시 새로 정리해야 한다.

역할:
- 차량별/기간별 가격 계산 기준
- 상세/목록/예약 확정가의 공통 계산 근거

핵심 요구:
- 차량별 가격을 재현 가능해야 한다.
- 기간이 바뀌면 같은 규칙으로 다시 계산 가능해야 한다.
- 최종 결제 직전에도 같은 함수/같은 테이블로 재검증 가능해야 한다.

권장 최소 구조:
- `car_prices`
  - `id`
  - `car_id`
  - `valid_from`
  - `valid_to`
  - `base_daily_price`
  - `weekend_daily_price` 또는 요일 규칙
  - `insurance_daily_price`
  - `discount_rule_json` 또는 할인 규칙 필드
  - `is_active`
  - `created_at`
  - `updated_at`

보조 구조 후보:
- `delivery_price_rules`
- `car_price_overrides`
- `holiday_pricing_rules`

### C. 예약 DB
새로 만드는 것이 맞다.

역할:
- IMS 예약 상태를 우리 시스템에서 조회 가능한 구조로 유지
- 기간 겹침 차단
- 가용 차량 계산
- 예약 확정가 저장
- 추후 변경/취소/동기화 상태 추적

권장 최소 구조:
- `reservations`
  - `id`
  - `ims_reservation_id` nullable
  - `car_id`
  - `status`
  - `start_at`
  - `end_at`
  - `pickup_option`
  - `delivery_region_id` nullable
  - `customer_name` nullable
  - `customer_phone` nullable
  - `search_session_id` nullable
  - `quoted_price_snapshot` jsonb
  - `confirmed_price_snapshot` jsonb
  - `sync_source` (`ims`, `local`, `manual`)
  - `synced_at`
  - `created_at`
  - `updated_at`

보조 구조 후보:
- `reservation_sync_jobs`
- `reservation_sync_logs`
- `reservation_status_history`

### D. search session
방식 A 서버 저장형으로 간다.

역할:
- 목록 응답 snapshot 저장
- 상세/새로고침/공유 대응
- 클라이언트 변조 방지

권장 구조:
- `search_sessions`
  - `id`
  - `search_snapshot` jsonb
  - `result_snapshot` jsonb
  - `expires_at`
  - `created_at`

또는 분리형:
- `search_sessions`
- `search_session_cars`

## 예약 DB를 IMS 준실시간 동기화 DB로 만드는 안 검토

## 판단
**적절하다. 다만 'IMS를 실시간 source of truth 로 직접 믿는 구조'가 아니라, '우리 예약 DB를 주 저장소로 두고 IMS를 지속 동기화하는 구조'가 더 안전하다.**

### 왜 적절한가
- 검색/상세/예약 확정이 IMS 응답 속도와 장애에 덜 흔들린다.
- 예약 중복 차단을 우리 DB 기준으로 먼저 할 수 있다.
- 장애 시 재동기화/복구가 가능하다.
- 추후 관리자/운영 화면과도 자연스럽게 연결된다.

### 주의할 점
- 완전 실시간이라고 가정하면 안 된다.
- 동기화 지연/실패를 고려해야 한다.
- 따라서 예약 생성 직전에는
  - 우리 DB 기준 재검증
  - 필요하면 IMS 최종 확인
  를 추가해야 한다.

### 권장 운영 원칙
- 평시 조회/가용성 계산은 우리 예약 DB 기준
- 예약 확정 직전은 최종 재검증
- IMS 동기화는 준실시간 pull/push + 실패 로그 관리

## partner 제거 이후의 기준 구조

### 검색
입력:
- `deliveryDateTime`
- `returnDateTime`
- `pickupOption`
- `driverAge`
- 지역 조건

처리:
1. 차량 DB에서 후보 추출
2. 예약 DB에서 해당 기간 겹침 제거
3. 가격 테이블로 차량별 가격 계산
4. 업체/딜리버리 정보 결합
5. 결과를 `searchSession` 으로 저장
6. 목록 응답 반환

### 상세
입력:
- `searchSessionId`
- `carId`

처리:
1. `searchSession` 조회
2. snapshot 안의 차량 정보 확인
3. 차량 DB 기준 상세 보강
4. 표시용 상세 렌더

원칙:
- partner 상세 재조회 없음
- 클라이언트가 보낸 가격/보험 신뢰 안 함

### 예약 확정
입력:
- `searchSessionId`
- `carId`
- 예약자 정보
- 결제 정보

처리:
1. `searchSession` 기준 검색 조건 복원
2. 예약 DB 기준 기간 겹침 재검사
3. 가격 테이블 기준 최종 가격 재계산
4. 확정가 snapshot 생성
5. 예약 저장
6. 필요 시 IMS 반영

### 종료 조건
- 특정 차량/기간의 예약 가능 여부를 우리 DB만으로 판정 가능

## Phase 4. 목록 API 전환
### 목적
검색 결과를 우리 계산 기준으로 생성한다.

### 작업
- 검색 조건 수신
- 차량 후보 조회
- 예약 DB로 가용성 필터
- 가격 계산

### 목적

### 작업

### 종료 조건
- 클라이언트 조작과 무관하게 서버 기준으로 확정 가능
- 확정가/예약 이력 추적 가능

## 보안 / 구조 검증 기준

### 통과 조건
- 목록에서 받은 가격을 클라이언트가 바꿔도 예약 확정가는 서버가 다시 계산한다.
- `searchSessionId` 없이 상세 핵심 정보가 복원되지 않는다.
- 다른 차량의 `carId` 를 임의 대입해도 세션 밖 차량은 조회되지 않는다.
- 예약 중복은 DB 레벨 또는 최소 서버 강제 검증으로 차단된다.
- IMS 동기화 지연이 있어도 우리 DB 기준으로 운영 가능하다.

### 위험 조건
- 가격을 프론트 값 그대로 저장
- 상세에서 클라이언트 state 만 신뢰
- 예약 중복 차단을 UI 로직에만 의존
- IMS 장애 시 검색/예약 전체가 멈춤
