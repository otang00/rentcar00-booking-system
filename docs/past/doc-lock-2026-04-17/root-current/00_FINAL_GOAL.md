# 00. FINAL GOAL

## 목적
`premove-clone`을 단순 외형 프로토타입이 아니라,
**실제 예약 서비스로 확장 가능한 사용자용 렌터카 예약 프론트**로 재정의한다.

현재 기준 최종 목표는 아래와 같다.
- 메인 페이지에서 검색 조건을 입력한다.
- 검색 조건 기준으로 가용 차량 목록을 본다.
- 차량을 클릭하면 해당 차량의 예약 페이지로 이동한다.
- 예약자 정보를 입력하고 결제를 진행한다.
- 결제 성공 후 예약이 저장되고, 이후 IMS 예약 반영까지 연결 가능해야 한다.

---

## 최종 제품 정의

### 사용자 관점
이 서비스는 아래 흐름을 가진다.
1. 메인에서 대여/반납 조건 입력
2. 조건에 맞는 차량 목록 확인
3. 차량 선택
4. 차량별 예약 상세 확인
5. 예약자 정보 입력
6. 약관 동의
7. 결제
8. 예약 완료

### 운영 관점
현재 구현 전략은 다음이다.
- 차량 조회는 **partner.premove.co.kr 검색 결과를 우리 서버가 받아 파싱/가공**한다.
- 프론트는 partner를 직접 호출하지 않는다.
- 예약/결제/저장은 우리 서버 흐름으로 구성한다.
- 향후 필요 시 partner 파싱 경로를 직접 IMS 연동 경로로 교체 가능해야 한다.

---

## 절대 기준

### 1. 메인 = 목록 시작점
- 메인 페이지는 단순 랜딩이 아니다.
- 검색 조건 입력 + 차량 목록 진입의 출발점이다.

### 2. 예약 페이지는 분리된다
- 목록에서 차량 클릭 시 차량별 예약 페이지로 이동한다.
- 예약 페이지는 선택 차량과 검색 조건을 이어받아야 한다.

### 3. 프론트 직박 금지
- partner/IMS 응답을 프론트에서 직접 때리지 않는다.
- 우리 서버가 조회하고 정리한 JSON만 프론트가 쓴다.

### 4. 현재는 MVP 현실성 우선
- 가장 빠른 경로는 partner 검색 결과 파싱 방식이다.
- 단, 구조는 나중에 교체 가능하게 설계한다.

### 5. 더미 구조 청산
- 이제부터 `mock` 중심 설계는 최종 방향이 아니다.
- 더미는 UI 보조용 fallback만 허용한다.

## 이번 프로젝트의 범위

### 포함
- 메인 검색 페이지
- 차량 목록 노출
- 차량 상세/예약 페이지
- 예약자 입력
- 약관 동의
- 결제 진입 흐름
- 예약 완료 화면

### 후속 단계 포함 가능
- 실제 결제 연동
- IMS 예약 생성
- 예약 저장/조회
- 실패 보상 처리

### 제외
- 관리자 백오피스
- 회원 시스템 대규모 구축
- 다업체 멀티테넌트 일반화
- 운영 통계 대시보드

## 성공 기준
다음 조건을 만족하면 방향이 맞다.
- 메인에서 검색 조건을 넣고 차량 목록을 실제처럼 볼 수 있다.
- 차량 클릭 시 해당 차량 예약 페이지로 자연스럽게 넘어간다.
- 예약 페이지가 정적 디자인이 아니라 실제 입력 흐름을 가진다.
- 데이터 소스가 partner 파싱 기반이더라도 프론트는 이를 모른다.
- 이후 결제/IMS 연결을 붙일 수 있는 구조가 된다.

---

## 비성공 기준
아래는 피해야 한다.
- 메인/목록/상세가 각각 따로 놀고 상태가 이어지지 않는 구조
- 프론트가 partner/IMS에 직접 의존하는 구조
- 페이지는 예쁘지만 데이터 흐름이 없는 구조
- 더미 필드명과 실제 필드명이 뒤섞여 교체가 어려운 구조
- 설계 문서는 많지만 실제 우선순위가 없는 상태

---

## 핵심 문장
이 프로젝트의 목표는
**partner 검색 결과를 안정적으로 받아 우리 서비스 UX로 재구성하고, 이후 결제와 예약 생성까지 확장 가능한 예약 서비스 뼈대를 완성하는 것**이다.
