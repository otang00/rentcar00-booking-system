- 예약시스템, 운영관리, 결제, 회원, 인증까지 확장될 가능성을 이름에 반영해야 한다.

- 향후 운영 어드민 분리 시 확장 후보: `rentcar00-admin`, `rentcar00-api`

## 2. 결제시스템 연동

### 목표
- 온라인 예약 완료 시 즉시 결제가 가능하도록 한다.
- 1차 목표는 가장 빠르고 안정적인 결제 도입이다.

### 권장 방향
- 1차: 토스페이먼츠 우선 검토
- 2차: 필요 시 네이버페이, 카카오페이 추가

### 판단기준
- 초기 연동 난이도
- 문서 품질과 테스트 환경
- 웹/모바일 브라우저 안정성
- 예약취소, 부분취소, 환불 처리 용이성
- 웹훅 처리 안정성

### 권장 구현 순서
1. 토스페이먼츠 기본 결제창 연동
2. 주문 생성, 결제 승인, 실패, 취소, 환불 플로우 구현
3. 웹훅 및 서버 검증 추가
4. 운영 테스트 완료 후 간편결제 추가 검토

### 검증기준
- 테스트 결제 성공, 실패, 취소 케이스 통과
- 주문번호 중복 방지
- 서버 승인 검증 통과
- 관리자 화면에서 결제상태 확인 가능

## 3. 로그인 및 회원 운영

### 목표
- 예약 이력 관리와 재예약 편의성을 위해 회원 체계를 도입한다.
- 단, 예약만 빠르게 받는 것이 우선이라면 비회원 예약도 유지한다.

### 권장 정책
- 비회원 예약 + 회원 예약 병행
- 회원가입 강제 대신 예약 후 계정 연동 가능 구조 권장

### 소셜 로그인 우선순위
1. 카카오 로그인
2. 네이버 로그인
3. 일반 휴대폰 또는 이메일 로그인
4. 토스 로그인은 별도 검토 대상

### 권장 구조
- 사용자 테이블은 내부 `user_id` 기준으로 운영
- 소셜 계정은 별도 `social_accounts` 테이블로 연결
- provider: `kakao` / `naver` / `email_phone`

### 실행단계
1. 회원정책 확정
2. 내부 계정 구조 설계
3. 카카오 로그인 연동
4. 네이버 로그인 연동
5. 비회원 예약 조회용 인증 방식 설계
6. 운영정책(탈퇴, 중복계정, 번호변경) 문서화

### 검증기준
- 같은 사용자가 소셜/일반 로그인을 섞어도 중복회원 폭증 없음
- 예약내역 조회 가능
- 로그인 실패 및 계정연동 충돌 처리 가능

### 검증기준

## 5. 휴대폰 번호 검증 + 운전면허 확인

### 판단 분리
- `번호 점유 확인`과 `실명 본인확인`은 다르다.
- `운전면허 진위 확인`도 별도 문제다.

### 권장 1차안
- 예약 단계: 휴대폰 인증번호(SMS OTP)로 번호 점유 확인
- 계약 또는 출고 단계: IMS 기반 운전면허 확인 프로세스 사용

### 이유
- 초기에 개발 복잡도와 실패율을 낮출 수 있다.
- 예약 전 단계에서 면허 진위확인까지 넣으면 이탈률이 높아질 수 있다.
- 실제 차량 인도 직전 확인이 실무에 더 맞다.

### 2차 확장안
- 필요 시 휴대폰 본인확인(PASS, 토스인증, NICE 계열) 추가
- 고위험 예약만 선별적으로 강화 인증 적용

### 검증기준
- 문자 인증 성공률 확인
- 번호 재사용 및 중복 악용 방지 정책 수립
- 예약 후 면허확인 미통과 시 예약상태 분기 가능

## 8. 최우선 선행 과제: 3개 시스템 연동 구조 확정

### 현재 확인된 사실
- 현재 `reservations` 는 순수 운영 예약 원장이 아니라 IMS 기준 upsert read model 성격이 강하다.
- 동기화는 `ims_reservation_id` 기준 upsert 이며 이번 배치에 내려오지 않은 기존 row는 삭제하지 않는다.
- `ims_reservations_raw` 는 실행마다 누적 저장되고, `reservations` 는 현재 상태성 read model 역할이다.
- 운영 데이터 외에 `shadow-seed` 테스트 데이터가 섞여 있다.
- 삭제 동기화(tombstone) 전략이 아직 없고 일부 과거 종료건이 남아 있을 수 있다.
- 고객용 조회에 필요한 공개 예약번호, 조회키, 상태이력이 아직 부족하다.
- `local-owned` 필드와 `sync-owned` 필드 분리가 없어 추후 자체 예약 생성 또는 수정 시 sync overwrite 위험이 있다.

### 현재 결론
- `reservations` 를 그대로 비회원 예약조회 및 자체 예약의 source of truth 로 사용하면 위험하다.
- 현재 테이블은 가용성 계산용 sync cache 또는 IMS read model 로 보는 것이 맞다.

### 바로 잠글 구조 원칙
1. `reservations` 는 계속 IMS sync read model 로 유지하거나 이름을 그 성격에 맞게 분리한다.
2. 고객 예약, 결제, 조회 기준 데이터는 별도 `local-owned` 테이블로 분리한다.
3. 테스트 데이터(`shadow-seed`)는 운영 조회에서 반드시 제외하고 가능하면 테이블 또는 환경 자체를 분리한다.
4. IMS에서 사라진 예약을 바로 삭제하지 말고 soft-state 기반 tombstone 전략을 둔다.

### 권장 테이블 분리안
#### A. sync-owned
- `reservations` 또는 `ims_sync_reservations`
- 목적: IMS 현재 상태 read model, 가용성 계산, 외부 데이터 캐시
- 주요 키: `ims_reservation_id`

#### B. local-owned
- `booking_orders`
- 목적: 우리 웹 예약의 기준 원장
- 주요 키: `booking_order_id`, `public_reservation_code`
- 보관: 고객정보, 결제상태, 우리 서비스 상태, 조회 노출 기준

#### C. lookup/security
- `booking_lookup_keys`
- 목적: 비회원 예약조회 인증용 키 저장
- 예: `booking_order_id`, `phone_hash`, `phone_last4`, `lookup_token`, `verified_at`

#### D. status/event history
- `reservation_status_events`
- 목적: 예약확정, 결제완료, 취소, 배차, 반납완료, 수동보정 이력

#### E. cross-system mapping
- `reservation_mappings`
- 목적: 우리 예약과 IMS 또는 기타 시스템 ID 매핑
- 예: `booking_order_id`, `ims_reservation_id`, `external_system_name`, `external_reservation_id`

### 먼저 결정할 것
1. 현재 `reservations` 를 유지할지, `ims_sync_reservations` 로 명확히 분리할지
2. 자체 예약 원장 테이블명을 `booking_orders` 로 할지
3. public 예약번호 규칙
4. 비회원 조회키 규칙
5. tombstone / archival 규칙
6. 테스트데이터 분리 방식

### 권장 실행 순서
1. 운영/테스트 데이터 분리
2. sync-owned / local-owned 역할 분리
3. mapping 테이블 도입
4. 비회원 조회용 공개 예약번호와 조회키 도입
5. 상태 이벤트 로그 도입
6. 이후 비회원 예약/결제 기능 구현
7. 그 다음 로그인/회원연동

### 현재 기준 실무 결론
지금 가장 먼저 할 일은 로그인이나 조회 UI가 아니라,
**IMS sync read model 과 우리 예약 원장을 분리하는 DB 구조 잠금**이다.
이게 끝나야 비회원 예약, 결제, 조회, 로그인 기능이 안 꼬인다.

## 9. 지금부터의 직접 실행 범위

### 바로 착수할 작업
1. 문서 체계를 이 문서 1개 기준으로 재정렬
2. DB 구조 설계 초안 작성
3. sync-owned / local-owned 경계 잠금
4. 예약 원장, 조회키, 상태이력, 매핑 구조를 먼저 확정

### 구현 전 선행 검증 질문
1. 우리 자체 예약 생성 시점은 결제 전인가 결제 완료 후인가
2. IMS 생성 시점은 결제 승인 직후인가 운영 확정 후인가
3. `public_reservation_code` 를 사람이 읽는 예약번호로 쓸지 여부
4. 비회원 조회 인증을 `예약번호 + 휴대폰` 조합으로 갈지, 토큰 방식 병행할지
5. 취소/환불 주도권을 우리 시스템이 가질지 IMS가 가질지

### 다음 문서 작업의 종료 조건
- DB 구조 설계안에서 테이블 책임이 겹치지 않음
- sync overwrite 위험 구간이 명확히 제거됨
- 비회원 예약조회 기능을 시작할 최소 스키마가 나옴
- 이후 결제 및 회원연동이 이 설계 위에 자연스럽게 올라갈 수 있음

## 10. DB 구조 설계 Phase 1 착수 기준

### 목적
- IMS sync read model 과 우리 예약 원장을 분리하는 최소 스키마를 먼저 잠근다.
- 비회원 예약조회, 결제, 회원연동이 모두 이 구조 위에서 확장되도록 한다.

### Phase 1 설계 범위
1. `sync-owned` 테이블 책임 확정
2. `local-owned` 예약 원장 책임 확정
3. 조회 인증용 키 저장 구조 확정
4. 상태 이벤트 이력 구조 확정
5. IMS 및 외부 시스템 매핑 구조 확정

### Phase 1 산출물
- 테이블별 역할 정의
- 필수 컬럼 목록
- PK, UK, FK 초안
- 상태 전이 기준 초안
- sync overwrite 방지 규칙
- 운영/테스트 데이터 분리 원칙

### Phase 1에서 잠글 최소 질문
1. `booking_orders` 생성 시점은 결제 전인지 결제 승인 직후인지
2. `public_reservation_code` 를 주문 생성 즉시 발급할지 결제 성공 후 발급할지
3. 비회원 조회 기본 인증을 `예약번호 + 휴대폰 뒤4자리` 로 시작할지
4. IMS 전송 실패 시 local 상태를 어떻게 보류할지
5. `shadow-seed` 를 별도 테이블로 뺄지 별도 환경으로 격리할지

### Phase 1 완료 기준
