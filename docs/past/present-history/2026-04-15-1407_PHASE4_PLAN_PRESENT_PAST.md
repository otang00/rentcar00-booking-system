# PHASE 4 PLAN — SEARCH RESULT ALIGNMENT

## Overview
Phase 4 목표는 DB 검색 결과를 partner 결과에 최대한 맞추는 것이다. Shadow 로그와 샘플 쿼리를 근거로 차이를 분석하고, 상태/시간/정렬/가격 순으로 보정한다. 아래 세부 페이즈는 순차적으로 진행하며 각 단계 완료 시 다음 단계로 이동한다.

---

## Phase 4-A — Diff 데이터 정착
- **목적:** shadow 로그 품질과 샘플 쿼리 세트를 검증해 보정에 사용할 데이터 기준 마련
- **주요 작업**
  1. `search_shadow_diffs`에서 최근 N건 추출
  2. Phase 0에서 잠근 샘플 쿼리 20~30건 재실행
  3. 차이 지표(카운트/순서/가격/상태) 대시보드 or 리포트 초안 작성
- **출력:** 보정 우선순위가 명확한 diff 리포트, 로그 수집 상태 체크리스트
- **종료 조건:** 우선순위 상위 5개 차이 유형 정의 + 예시 데이터 확보

## Phase 4-B — 상태/활성 조건 보정
- **목적:** partner 노출 차량과 DB 노출 차량의 상태 정의 차이를 해소
- **주요 작업**
  1. partner → DB 상태 매핑 테이블 작성
  2. `cars.status`, `reservations.status` 의 blocking 규칙 재검증 (특히 `대기중/배차중/점검/불가`)
  3. Shadow diff에서 "missingInDb" 중 상태 문제로 분류된 케이스 재현
  4. `filterAvailableCars` 규칙 수정 계획 작성 (실제 수정은 Phase 4 실행 승인 후)
- **출력:** 상태 매핑 표, 수정 TODO 리스트, 영향 분석
- **종료 조건:** 상태 차이로 발생한 diff 케이스가 모두 분류되어 있고, 수정안이 준비됨

## Phase 4-C — 시간/버퍼 로직 보정
- **목적:** pickup/return 시간대 처리, 버퍼, 딜리버리 오프셋 차이 조정
- **주요 작업**
  1. partner API 시간 처리 규칙 재수집 (평일/주말/딜리버리)
  2. `buildSearchWindow`, `expandRange` 버퍼 값 검증 및 대안 시뮬레이션
  3. Shadow diff의 시간 겹침 케이스 재현해 버퍼 값 조정안 도출
- **출력:** 시간 관련 파라미터 표 + 테스트 전략
- **종료 조건:** 제안 버퍼/시간 계산안이 문서화되고, 조정 시 영향 범위가 파악됨

## Phase 4-D — 정렬/가중치 보정
- **목적:** DB 결과 정렬 순서를 partner 순서와 맞춤
- **주요 작업**
  1. partner 정렬 기준(가격, 차량 등급, 노출 우선순위) 재확인
  2. 현재 `dbSearchService` 정렬 로직 점검
  3. `orderVariance` 큰 샘플에 대해 중간 정렬 스텝 비교
- **출력:** 정렬 규칙 차이 분석, 우선 적용해야 할 가중치 목록
- **종료 조건:** 정렬 차이 주요 원인이 문서화되고, 수정 계획이 확정됨

## Phase 4-E — 가격 보정
- **목적:** partner와 동일한 가격 규칙 (할인, 프로모션, VAT 등)을 DB 계산에 반영
- **주요 작업**
  1. partner 가격 산식 재수집 (기본 요금, 옵션, 할인)
  2. Shadow diff의 `priceDiffs` 샘플 상세 비교
  3. DB 가격 계산 모듈 설계 (Phase 5 전환 대비)
- **출력:** 가격 계산 파이프라인 설계 초안, 필요한 데이터 항목 목록
- **종료 조건:** 가격 차이 보정안이 정리되고, 구현 단계로 넘길 수 있는 사양 확정

## Phase 4-F — 통합 검증 & 전환 게이트
- **목적:** 각 보정이 반영된 후 결과 차이가 목표 기준 이하인지 확인하고 Phase 5 전환 준비
- **주요 작업**
  1. Phase 0 샘플 세트 재검증
  2. Shadow 로그 → 집계 리포트 자동화(필요 시)
  3. Phase 5 전환 체크리스트 초안 작성
- **출력:** 샘플별 diff Before/After 비교표, 전환 readiness 리포트
- **종료 조건:** 허용 오차 내 결과 확보 + 전환 플래그 배포 계획 수립

---

## 준비 메모
- 모든 단계는 “분석/계획”과 “구현”을 분리한다. 현재는 분석/문서화까지만 진행.
- 실 데이터 추출이나 코드 수정은 사장님 승인 후 Phase 4 각 단계별 실행으로 전환한다.
- Diff 로그 활용을 위해 `search_shadow_diffs` 쿼리 템플릿, 샘플 요청 목록을 준비해 둔다 (추가 지시에 따라 작성).
