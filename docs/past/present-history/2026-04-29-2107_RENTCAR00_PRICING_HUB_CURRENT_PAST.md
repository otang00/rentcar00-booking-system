# RENTCAR00 price hub current

Last updated: 2026-04-29

이 문서는 RENTCAR00 가격표 허브 작업의 현재 기준 문서다.
이제 active 구현 축은 예약 연동이 아니라 **가격표 허브 DB를 만들고, 수정 가능한 기준 가격을 다시 각 채널/내부 알고리즘으로 뿌리는 구조**다.

---

## 1. 이번 단계 목표

### 한 줄 정의
이번 작업은 **IMS 중심 가격 구조를 본체로 유지하고, 찜카에서 필요한 요소만 확장 추가한 뒤 운영 기준으로 수정하고 배포 출력으로 재사용하는 구조**를 만드는 것이다.

### 목표
- IMS 기준 그룹/가격 구조를 메인 본체로 잠근다.
- 찜카 가격표를 IMS 축에 연결하는 매핑 구조를 만든다.
- 찜카 반영에 필요한 확장 요소만 별도로 추가한다.
- 운영자가 IMS 중심 허브 기준값을 수정할 수 있게 한다.
- 수정 결과를 내부 검색/노출/외부 재배포용 출력으로 다시 뿌릴 수 있게 한다.

### 성공 기준
- 가격 source of truth가 파일이 아니라 DB 기준으로 잠긴다.
- IMS / 찜카 입력 가격이 같은 메인 그룹 축으로 비교 가능하다.
- 허브에서 수정한 값이 계산 결과에 일관되게 반영된다.
- 원본 입력값 / 매핑값 / 최종 출력값이 분리되어 추적 가능하다.
- 이후 채널 추가 시에도 같은 허브 구조에 붙일 수 있다.

### 이번 단계 비포함
- 결제 시스템 도입
- 예약 상태 동기화 재설계
- IMS 예약 생성/취소 플로우 재구축
- 찜카 disable_time 후속 동기화 재작업
- 관리자 UI 완성형 설계

## 1-1. 네이밍 규칙

이번 작업부터 새 체계는 아래 규칙으로 잠근다.

### 핵심 규칙
- `PRICE` 단독 명칭은 신규 체계 이름으로 쓰지 않는다.
- 신규 체계 공통 명칭은 **`RENTCAR00_PRICING_HUB`** 로 통일한다.
- 사용자 화면/문서/구현 설명에서도 가능하면 `가격 허브` 대신 **`통합 요금체계`** 또는 **`PRICING_HUB`** 로 부른다.

### 이유
- `PRICE` 와 `PRICING` 이 너무 비슷해 기존 구조와 신규 구조를 헷갈리기 쉽다.
- 새 체계는 기존 price 테이블 1개가 아니라 그룹/기간/요율/예외/preview/publish 를 포함한 더 큰 구조다.

### 적용 규칙
- 문서 제목/섹션: `RENTCAR00_PRICING_HUB_*`
- 신규 테이블 prefix 후보: `pricing_`
- 신규 패널/화면 명칭: `통합 요금 관리` 또는 `PRICING_HUB`
- 기존 `price_policies` 같은 레거시 이름은 유지하되, 신규 개념명으로 재사용하지 않는다.

## 2. 기준점

### 이미 끝난 축
완료된 구현 축은 아래다.
- IMS 예약 sync 적재
- 찜카 disable_time 후속 동기화
- 예약/운영 패널 기초 반영

이 축은 이제 active 구현이 아니라 **참고 가능한 과거 자산**이다.

### 이번 축의 핵심 차이
예약 연동은 시간대 차단 문제였다.
이번 가격 허브는 **가격 원천 통합 + 메인 매핑 + 수정 후 재배포** 문제다.

즉 이번에는 아래 3개를 먼저 잠가야 한다.
1. 메인 식별 축이 무엇인지
2. 입력 가격을 어떤 단위로 저장할지
3. 수정 후 어떤 출력 계약으로 뿌릴지

## 3. 메인 허브 구조 원칙

### 원칙 1 — 허브 기준 축은 IMS group 축 고정
현재 기준으로 가격 계산의 canonical 축은 아래 흐름으로 잠근다.
- `source_group_id`
- `ims_group_id`
- `policy`

즉 1차 구현은 별도 독립 canonical 축보다 **IMS group 축 중심 확장형**으로 간다.

### 원칙 2 — 원본 / 매핑 / 운영값 / 출력값 분리
한 테이블에 다 섞지 않는다.
최소한 아래 층위는 분리한다.
- 원본 입력값
- 메인 매핑값
- 운영 수정값
- 최종 배포/계산 출력값

### 원칙 3 — 파일은 입력물, DB가 기준
엑셀/CSV는 업로드 입력물이다.
운영 기준 가격은 DB 허브가 source of truth 여야 한다.

### 원칙 4 — 채널명보다 차량/그룹 식별이 먼저
가격 숫자보다 먼저 잠글 것은 아래다.
- IMS 쪽 그룹 식별자
- 찜카 쪽 차량/상품 식별자
- 같은 대상을 묶는 메인 mapping key

### 원칙 5 — 수정 가능성과 재생성 가능성을 같이 가져간다
운영자가 값 수정은 할 수 있어야 한다.
동시에 원본 입력부터 출력까지 재계산 가능한 구조여야 한다.

## 5. 핵심 질문

이번 문서 기준으로 먼저 답해야 할 질문은 아래다.

1. 메인 key 는 `ims_group_id` 단독인지, 별도 hub group id 인지
2. 찜카 가격표는 차량 단위인지 그룹 단위인지
3. 운영 수정은 절대값인지, 가감값인지, 정책 테이블인지
4. 출력은 검색 API 직접 참조인지, 별도 materialized 결과인지
5. 수동 수정 후 원본 재업로드 시 merge 규칙을 어떻게 둘지

이 질문이 잠기기 전까지는 세부 구현을 넓히지 않는다.

## 6. 과거 구현 자산 참고

### IMS 예약 연동에서 남겨야 할 참고
- 인증 기본 경로: `POST https://api.rencar.co.kr/auth`
- 인증 방식: `password = sha256(plainPassword).hex()` 후 JWT 발급
- 예약 조회 API: `GET /v2/company-car-schedules/reservations`
- 예약 취소 API: `POST /v2/company-car-schedules/delete`
- 생성은 직접 API가 아니라 UI 자동화 검증 이력만 있음

### 찜카 연동에서 남겨야 할 참고
- 로그인: `POST /login`
- 차량검색: `GET /vehicle/vehicle/paging`
- 차량상세: `POST /vehicle/vehicle/detail/{pid}`
- 휴차시간 조회: `GET /vehicle/vehicle/{vehiclePid}/disable_time`
- 휴차시간 생성: `PUT /vehicle/disable_time`
- 휴차시간 삭제: `DELETE /vehicle/disable_time`

### 왜 참고로 남기나
이번 작업은 가격 허브가 active지만,
향후 가격 출력이나 채널 반영 연결 시 외부 채널 식별/호출 기준이 다시 필요해질 수 있다.
그래서 **구현은 과거로 내리되 API 사실관계는 active 참고로 유지**한다.
