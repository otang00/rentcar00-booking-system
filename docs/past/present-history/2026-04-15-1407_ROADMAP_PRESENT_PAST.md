# ROADMAP

## 목적
현재 partner 프록시 기반 검색 결과를 **우리 DB 기반 검색 결과로 교체**하되,
사용자 체감상 결과는 지금과 최대한 같게 유지한다.

## 기준점
- 시작 기준 commit: `31f55f2`
- 의미: **DB 연결 전 포인트**
- 현재 검색 결과 source: partner 파싱
- 현재 예약 source: IMS 동기화로 적재된 `reservations`

## 최종 목표
검색 페이지가 아래 구조로 동작하게 만든다.

- 프론트 검색 페이지
- 우리 서버 검색 API
- 우리 DB(`cars + reservations`) 기반 계산
- 기존과 최대한 동일한 응답 shape 반환

## 비목표
이번 로드맵 범위에는 아래를 넣지 않는다.
- 관리자 페이지 제품화
- 운영자 권한 시스템
- 본격 로그인 체계
- 상세/예약 확정 전체 재구축

  - 평일/주말
  - 단기/월렌트
  - 딜리버리 유/무
  - 차량 적은 시간대/많은 시간대

### 종료 조건

- 샘플 비교 세트 확보됨

## Phase 1. DB 검색 read model 설계
### 목적
우리 DB만으로 검색 후보 차량을 계산할 수 있게 만든다.

### 할 일
- `cars` 를 검색용 차량 기준 테이블로 확정
- `reservations` 에서 blocking 상태 규칙 확정
- 예약 겹침 판단 규칙 확정
- 검색 결과 계산에 필요한 최소 read model 정의
  - 사용 가능 차량 목록
  - 예약 겹침 제외
  - 기본 정렬 키
  - 노출용 차량 필드

### 종료 조건
- DB 기준으로 후보 차량 목록을 산출하는 서버 함수 설계 완료

---

## Phase 2. DB 검색 서비스 구현
### 목적
기존 endpoint를 유지한 채, 내부에 DB 검색 엔진을 만든다.

### 구현 원칙
- 프론트 endpoint는 가능하면 유지
- 새 공개 API를 무리하게 늘리지 않음
- 서버 내부에 `dbSearchService` 계층을 둠

### 할 일
- `dbSearchService` 구현
- `cars + reservations` 조회 로직 작성
- 시간 겹침 / blocking 상태 / 기본 필터 계산
- 기존 검색 응답 shape로 매핑

### 종료 조건
- 단독 실행 가능한 DB 검색 결과가 나옴
- 아직 사용자 응답은 기본적으로 기존 partner 결과 유지

---

## Phase 3. shadow mode 비교
### 목적
partner 결과와 DB 결과를 동시에 계산해 차이를 수집한다.

### 할 일
- 기존 검색 API 내부에서
  1. partner 결과 계산
  2. DB 결과 계산
  3. 현재 응답은 기존 결과 반환
  4. 차이 로그 저장
- diff 항목 정의
  - 결과 차량 수
  - 차량 id 목록
  - 순서 차이
  - 가격 차이
  - 제외 사유 차이

### 종료 조건
- 샘플 쿼리 기준 차이 패턴 수집 완료
- 어디가 다른지 데이터로 설명 가능

## Phase 5. 검색 전환
### 목적
검색 페이지를 DB 기준으로 전환한다.

### 할 일
- feature flag 도입
- 내부/preview에서 우선 검증
- 문제 없으면 DB 결과를 기본값으로 전환
- 롤백 시 partner 결과로 즉시 복귀 가능하게 유지

### 종료 조건
- 검색 페이지 기본 source가 DB로 전환됨
- 전환 후에도 기존 수준의 결과 일관성 유지
