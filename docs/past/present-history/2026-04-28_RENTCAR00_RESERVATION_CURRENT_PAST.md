## 1. 프로젝트명 정리

### 현재 문제
- `premove-clone` 은 외부용 이름으로 부적절하고 실제 사업 목적과 맞지 않는다.
- 예약시스템, 운영관리, 결제, 회원, 인증까지 확장될 가능성을 이름에 반영해야 한다.

### 현재 기준
- 대외 서비스명: 빵빵카(주) 예약시스템
- 저장소 및 개발 프로젝트명: `rentcar00-reservation`
- 차후 연결 도메인명: `rentcar00.com`
- 향후 운영 어드민 분리 시 확장 후보: `rentcar00-admin`, `rentcar00-api`

### 실행단계
1. 이름 확정
2. 코드, 환경변수, 패키지명, 문서 내 기존 명칭 전수 검색
3. 저장소명 및 프로젝트 내 display name 변경
4. OAuth 리다이렉트 URL, 결제 콜백 URL, 배포변수 이름 점검
5. QA 후 문서 일괄 정리

### 검증기준
- 화면, 문서, 환경변수, 로그에 `premove-clone` 잔존 0건
- 로그인, 결제, 웹훅, 리다이렉트 URL 정상 동작

## 2. 결제시스템 연동

### 목표
- 온라인 예약 완료 시 즉시 결제가 가능하도록 한다.
- 1차 목표는 가장 빠르고 안정적인 결제 도입이다.

### 권장 방향
- 1차: 토스페이먼츠 우선 검토
- 2차: 필요 시 네이버페이, 카카오페이 추가

### 판단기준
- 초기 연동 난이도
- 문서 품질과 테스트 환경
- 웹/모바일 브라우저 안정성
- 예약취소, 부분취소, 환불 처리 용이성
- 웹훅 처리 안정성

### 현재 취소 / 환불 운영 기준
- 온라인에서 가능한 것은 `예약 취소 요청` 까지만이다.
- 예약 취소 시 화면에는 환불 안내 규정만 출력한다.
- 고객에게는 `담당자 확인 후 환불 처리 예정` 이라고만 고지한다.
- 결제 취소 API 자동 호출, 실시간 환불금액 자동 계산, 자동 환불 완료 처리는 1차 범위에서 제외한다.
- 실제 환불 실행은 담당자 수동 처리로 운영한다.
- 시스템은 취소 요청 접수와 내부 처리 상태 기록에 집중한다.

### 주문 금액 스냅샷 기준
- 결제 1차 작업에서는 쿠폰/포인트 스키마를 먼저 도입하지 않는다.
- 대신 `booking_orders.pricing_snapshot` 에 최소 금액 구성요소를 저장한다.
- 현재 잠그는 최소 저장 필드는 아래 4개다.
  - `rentalAmount`
  - `insuranceAmount`
  - `deliveryAmount`
  - `finalAmount`
- 이 4개는 주문 생성 시점 기준 금액 스냅샷이며, 이후 쿠폰 도입 시 확장 기준점으로 사용한다.
- 별도 정규 컬럼 분리는 쿠폰 workstream 에서 다시 판단한다.

### 취소 시 사용자 고지 기준
- 취소 가능 여부와 환불 안내 규정은 화면에서 보여준다.
- 환불예정금액을 시스템이 확정 금액처럼 노출하지 않는다.
- 안내 문구는 `환불 규정에 따라 담당자 확인 후 처리됩니다` 를 기준으로 한다.
- 취소 완료 직후에도 `환불 완료` 로 표시하지 않고 `환불 대기` 또는 동등한 운영 상태로 처리한다.

### 운영 처리 기준
- 예약 취소 접수 후 `payment_status` 는 `refund_pending` 으로 둔다.
- 실제 담당자 확인 및 환불 완료 후에만 `payment_status=refunded` 로 변경한다.
- 자동 환불 실패 처리 분기는 당장 만들지 않고, 수동 처리 전제를 문서 기준으로 잠근다.
- 환불 금액 판단은 약관/운영 정책 기준으로 담당자가 최종 확정한다.

### 권장 구현 순서
1. 토스페이먼츠 기본 결제창 연동
2. 주문 생성, 결제 승인, 실패, 취소 요청 접수 플로우 구현
3. 취소 시 환불 규정 안내 문구와 `refund_pending` 상태 처리 구현
4. 웹훅 및 서버 검증 추가
5. 운영 테스트 완료 후 간편결제 추가 검토

### 검증기준
- 테스트 결제 성공, 실패, 취소 요청 케이스 통과
- 주문번호 중복 방지
- 서버 승인 검증 통과
- 취소 시 자동 환불 완료가 아니라 환불 규정 안내 + 환불 대기 상태가 노출됨
- 관리자 화면에서 결제상태 확인 가능

## 3. 로그인 및 회원 운영

### 목표
- 예약 이력 관리와 재예약 편의성을 위해 회원 체계를 도입한다.
- 단, 예약만 빠르게 받는 것이 우선이라면 비회원 예약도 유지한다.

### 권장 정책
- 비회원 예약 + 회원 예약 병행
- 회원가입 강제 대신 예약 후 계정 연동 가능 구조 권장

### 로그인 구현 우선순위
1. 이메일 + 비밀번호 로그인
2. 비밀번호 재설정 링크
3. 카카오 로그인
4. 네이버 로그인
5. 일반 휴대폰 인증 확장
6. 토스 로그인은 별도 검토 대상

### 권장 구조
- 인증 기준은 Supabase `auth.users.id` 로 운영
- 서비스용 사용자 정보는 별도 `profiles` 테이블로 연결
- 소셜 계정/provider 확장은 로그인 current 문서 기준으로 분리 관리한다.

### 실행단계
1. 회원정책 확정
2. 내부 계정 구조 설계
3. 이메일 + 비밀번호 로그인 연동
4. 비밀번호 재설정 경로 구현
5. 카카오 로그인 연동
6. 비회원 예약 조회용 인증 방식 설계
7. 운영정책(탈퇴, 중복계정, 번호변경) 문서화

### 검증기준
- 같은 사용자가 소셜/일반 로그인을 섞어도 중복회원 폭증 없음
- 예약내역 조회 가능
- 로그인 실패 및 계정연동 충돌 처리 가능

## 4. 주소 검색 + 지도 표시

### 목표
- 상세주소 입력 전 주소 검색 팝업 또는 검색창을 제공한다.
- 선택한 주소가 지도에서 바로 보이게 한다.

### 권장 방향
- 다음 우편번호 서비스 + 카카오맵 조합 우선
- 주소 선택 후 좌표 변환하여 지도에 마커 표시

### 구현 포인트
- 기본주소 / 상세주소 분리 저장
- 좌표(`lat`, `lng`) 저장
- 주소 수정 시 지도 즉시 갱신
- 모바일에서도 검색/지도 UI가 깨지지 않게 처리

### 검증기준
- 주소검색 성공
- 주소 선택 후 지도 표시 성공
- 저장된 주소 재조회 시 동일 위치 재표시 성공

## 5. 휴대폰 번호 검증 + 운전면허 확인

### 판단 분리
- `번호 점유 확인`과 `실명 본인확인`은 다르다.
- `운전면허 진위 확인`도 별도 문제다.

### 권장 1차안
- 예약 단계: 휴대폰 인증번호(SMS OTP)로 번호 점유 확인
- 계약 또는 출고 단계: IMS 기반 운전면허 확인 프로세스 사용

### 이유
- 초기에 개발 복잡도와 실패율을 낮출 수 있다.
- 예약 전 단계에서 면허 진위확인까지 넣으면 이탈률이 높아질 수 있다.
- 실제 차량 인도 직전 확인이 실무에 더 맞다.

### 2차 확장안
- 필요 시 휴대폰 본인확인(PASS, 토스인증, NICE 계열) 추가
- 고위험 예약만 선별적으로 강화 인증 적용

### 검증기준
- 문자 인증 성공률 확인
- 번호 재사용 및 중복 악용 방지 정책 수립
- 예약 후 면허확인 미통과 시 예약상태 분기 가능

## 8. 최우선 선행 과제: 3개 시스템 연동 구조 확정

### 현재 확인된 사실
- 현재 `reservations` 는 순수 운영 예약 원장이 아니라 IMS 기준 upsert read model 성격이 강하다.
- 동기화는 `ims_reservation_id` 기준 upsert 이며 이번 배치에 내려오지 않은 기존 row는 삭제하지 않는다.
- `ims_reservations_raw` 는 실행마다 누적 저장되고, `reservations` 는 현재 상태성 read model 역할이다.
- 운영 데이터 외에 `shadow-seed` 테스트 데이터가 섞여 있다.
- 삭제 동기화(tombstone) 전략이 아직 없고 일부 과거 종료건이 남아 있을 수 있다.
- 고객용 조회에 필요한 공개 예약번호, 조회키, 상태이력이 아직 부족하다.
- `local-owned` 필드와 `sync-owned` 필드 분리가 없어 추후 자체 예약 생성 또는 수정 시 sync overwrite 위험이 있다.

### 현재 결론
- `reservations` 를 그대로 비회원 예약조회 및 자체 예약의 source of truth 로 사용하면 위험하다.
- 현재 테이블은 가용성 계산용 sync cache 또는 IMS read model 로 보는 것이 맞다.

### 바로 잠글 구조 원칙
1. `reservations` 는 계속 IMS sync read model 로 유지하거나 이름을 그 성격에 맞게 분리한다.
2. 고객 예약, 결제, 조회 기준 데이터는 별도 `local-owned` 테이블로 분리한다.
3. 테스트 데이터(`shadow-seed`)는 운영 조회에서 반드시 제외하고 가능하면 테이블 또는 환경 자체를 분리한다.
4. IMS에서 사라진 예약을 바로 삭제하지 말고 soft-state 기반 tombstone 전략을 둔다.

### 권장 테이블 분리안
#### A. sync-owned
- `reservations` 또는 `ims_sync_reservations`
- 목적: IMS 현재 상태 read model, 가용성 계산, 외부 데이터 캐시
- 주요 키: `ims_reservation_id`

#### B. local-owned
- `booking_orders`
- 목적: 우리 웹 예약의 기준 원장
- 주요 키: `booking_order_id`, `public_reservation_code`
- 보관: 고객정보, 결제상태, 우리 서비스 상태, 조회 노출 기준
- 로그인 연동 기준:
  - 예약 원장은 계속 `booking_orders` 다.
  - 1차 구현은 `booking_orders.user_id nullable` 기준으로 진행한다.
  - 로그인 회원 예약만 `user_id` 연결 가능하고 비회원 예약은 `user_id = null` 로 유지한다.
  - 기존 비회원 예약의 회원 귀속은 사후 연결 정책으로만 처리한다.

#### B-2. member-owned
- `profiles`
- 목적: 서비스용 회원 기본정보 보관
- 기준키: `id = auth.users.id`
- 최소 역할: 이름, 연락처, 동의 상태, 생성/수정 시각 보관
- 원칙: 인증 원본은 Supabase Auth 이고 서비스용 표시 데이터만 `profiles` 에 둔다.
- 현재 상태: 테이블 추가 완료
- 현재 최소 컬럼: `id`, `email`, `name`, `phone`, `marketing_agree`, `created_at`, `updated_at`

#### C. lookup/security
- `booking_lookup_keys`
- 목적: 비회원 예약조회 인증용 키 저장
- 예: `booking_order_id`, `phone_hash`, `phone_last4`, `lookup_token`, `verified_at`

#### D. status/event history
- `reservation_status_events`
- 목적: 예약확정, 결제완료, 취소, 배차, 반납완료, 수동보정 이력

#### E. cross-system mapping
- `reservation_mappings`
- 목적: 우리 예약과 IMS 또는 기타 시스템 ID 매핑
- 예: `booking_order_id`, `ims_reservation_id`, `external_system_name`, `external_reservation_id`

### 먼저 결정할 것
1. 현재 `reservations` 를 유지할지, `ims_sync_reservations` 로 명확히 분리할지
2. 자체 예약 원장 테이블명을 `booking_orders` 로 할지
3. `profiles` 최소 컬럼과 생성 시점
4. public 예약번호 규칙
5. 비회원 조회키 규칙
6. tombstone / archival 규칙
7. 테스트데이터 분리 방식

### 권장 실행 순서
1. 운영/테스트 데이터 분리
2. sync-owned / local-owned 역할 분리
3. `profiles` 기준 잠금
4. mapping 테이블 도입
5. 비회원 조회용 공개 예약번호와 조회키 도입
6. 상태 이벤트 로그 도입
7. 이후 비회원 예약/결제 기능 구현
8. 그 다음 이메일 + 비밀번호 로그인/회원연동
9. 이후 카카오 로그인 확장

### 현재 기준 실무 결론
지금 가장 먼저 할 일은 로그인이나 조회 UI가 아니라,
**IMS sync read model 과 우리 예약 원장을 분리하는 DB 구조 잠금**이다.
이게 끝나야 비회원 예약, 결제, 조회, 로그인 기능이 안 꼬인다.

### 다음 진행 준비 메모
- 다음 실제 연결 작업은 예약 생성 시 로그인 세션이 있으면 `booking_orders.user_id` 를 저장하는 것이다.
- 이때도 비회원 예약 생성과 비회원 예약조회 구조는 그대로 유지한다.
- 즉, 회원 연결은 `booking_orders.user_id` 로 추가하고, guest lookup 축은 `booking_lookup_keys` 기반을 유지한다.
- 기존 비회원 예약을 회원 계정으로 귀속시키는 작업은 별도 후속 phase 로 남긴다.

## 10. DB 구조 설계 Phase 1 착수 기준

### 목적
- IMS sync read model 과 우리 예약 원장을 분리하는 최소 스키마를 먼저 잠근다.
- 비회원 예약조회, 결제, 회원연동이 모두 이 구조 위에서 확장되도록 한다.

### Phase 1 설계 범위
1. `sync-owned` 테이블 책임 확정
2. `local-owned` 예약 원장 책임 확정
3. 회원용 `profiles` 책임과 생성 시점 확정
4. 조회 인증용 키 저장 구조 확정
5. 상태 이벤트 이력 구조 확정
6. IMS 및 외부 시스템 매핑 구조 확정

### Phase 1 산출물
- 테이블별 역할 정의
- 필수 컬럼 목록
- PK, UK, FK 초안
- `auth.users.id` 와 앱 테이블 연결 기준
- 상태 전이 기준 초안
- sync overwrite 방지 규칙
- 운영/테스트 데이터 분리 원칙

### Phase 1에서 잠글 최소 질문
1. `booking_orders` 생성 시점은 결제 전인지 결제 승인 직후인지
2. `profiles` 는 최초 로그인 성공 시 upsert 할지 별도 onboarding 시점에 만들지
3. `public_reservation_code` 를 주문 생성 즉시 발급할지 결제 성공 후 발급할지
4. 비회원 조회 기본 인증을 `예약번호 + 휴대폰 뒤4자리` 로 시작할지
5. IMS 전송 실패 시 local 상태를 어떻게 보류할지
6. `shadow-seed` 를 별도 테이블로 뺄지 별도 환경으로 격리할지

### Phase 1 완료 기준
- 예약 원장 테이블 1개가 source of truth 로 명확히 잠김
- `reservations` 또는 `ims_sync_reservations` 가 sync cache 성격으로 분리됨
- 회원용 `profiles` 테이블 책임과 생성 시점이 잠김
- `booking_orders.user_id nullable` 기준이 잠김
- 비회원 예약조회에 필요한 최소 키 구조가 정의됨
- 회원 연결 이후에도 비회원 예약조회 구조가 유지됨
- 결제 성공, 취소 요청, 환불 대기, 환불 완료, IMS 반영 실패 시 상태 분기 규칙이 초안 수준으로라도 고정됨
