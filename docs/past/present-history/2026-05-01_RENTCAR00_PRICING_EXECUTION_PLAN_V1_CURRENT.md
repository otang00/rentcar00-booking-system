# RENTCAR00 pricing execution plan v1 current

Last updated: 2026-05-01

이 문서는 가격 개편 실행 계획을 3개 phase 기준으로 다시 정리한다.
복잡한 신규 구조 설계보다, 지금 구조를 최대한 유지하면서 필요한 부분만 추가하는 방향을 기준으로 한다.

---

## 1. 실행 목표

큰 phase 는 3개다.

1. `15일`, `15일 데일리`, `30일`, `30일 데일리` 컬럼 추가
2. 운영 페이지에 운영파라미터 추가
3. 계산식 재정립

이 문서의 목적은 이 3개 phase 의 범위, 대상, 종료 조건을 실행 기준으로 명확히 하는 것이다.

---

## 2. 현재 기준점

### 2-1. 현재 장기 source
확인된 monthly source 필드:
- `d15_total_cost`
- `d15_daily_cost`
- `m1_total_cost`
- `m1_daily_cost`

참고 문서:
- `docs/present/2026-05-01_RENTCAR00_PRICING_HUB_MONTHLY_SOURCE_CURRENT.md`

### 2-2. 현재 단기 계산 로직
핵심 파일:
- `server/search-db/pricing/calculateGroupPrice.js`

현재 사실:
- 버킷은 총 대여시간 기준
- `billableDays = ceil(totalHours / 24)`
- 주중/주말 판정은 `startAt + 24h*n` 시작요일 기준

즉 현재는 실제 포함 날짜 기준 계산이 아니다.

### 2-3. 현재 DB/관리 구조
기존 기반:
- `price_policies`
- `price_policy_groups`
- `pricing_hub_periods`
- `pricing_hub_rates`
- `pricing_hub_previews`
- `pricing_hub_preview_items`

기준:
- 기존 구조는 최대한 유지
- 필요한 필드와 운영 입력만 최소 추가

---

## 3. Phase 1. 장기 4개 컬럼 추가

### 목적
장기 계산에 필요한 공식 저장 칸을 먼저 만든다.

### 해야 할 일
1. 장기 4개 컬럼 추가 대상 테이블 확정
2. migration 작성
3. 기존 조회/편집 경로 영향 확인

### 추가 컬럼
- `d15_total_cost`
- `d15_daily_cost`
- `m1_total_cost`
- `m1_daily_cost`

### 확인할 것
- 어느 테이블에 붙일지
- 기존 허브 편집 흐름과 충돌 없는지
- null 허용/기본값 정책

### 종료 조건
- DB에 장기 4개 컬럼이 추가된다.
- 코드에서 읽고 저장할 위치가 분명하다.

---

## 4. Phase 2. 운영 페이지에 운영파라미터 추가

### 목적
운영자가 기준값을 정렬해서 직접 입력/조정할 수 있게 한다.

### 해야 할 일
1. 가격조정 페이지에 운영파라미터 입력 영역 추가
2. 기준가격 기준 정렬 지원
3. 주중 %, 주말 %, 월차 관련 값 입력 지원
4. 미리보기 또는 반영 전 확인 흐름 추가

### 최소 입력 항목
- 기준가격
- 주중 %
- 주말 %
- 월차 할인 관련 값
- 필요 최소 단기 계수

### 원칙
- 운영파라미터는 공식 확정 가격이 아니다.
- 운영자가 조정하는 입력 기준값이다.
- 반영 전 확인 절차가 있어야 한다.

### 종료 조건
- 운영 페이지에서 기준가격 순 정렬이 된다.
- 운영자가 핵심 파라미터를 입력/수정할 수 있다.

---

## 5. Phase 3. 계산식 재정립

### 목적
단기/장기 계산을 운영 기대값과 맞춘다.

### 5-1. 단기 계산
해야 할 일:
1. 시작요일 기준 계산 제거
2. 실제 포함 과금일 기준 주중/주말 분리
3. 기존 버킷 구조 유지

핵심 변경점:
- 기존: `startAt + 24h*n` 시작요일 기준
- 변경: 실제 포함된 과금일 기준

### 5-2. 장기 계산
해야 할 일:
1. 장기 4개 필드 기준 계산 구현
2. 15일 이상 30일 미만 계산식 반영
3. 30일 이상 계산식 반영

핵심식:
- `min(d15_total_cost + ((days - 15) × d15_daily_cost), m1_total_cost)`
- `total = (months × m1_total_cost) + remain_cost`

### 5-3. 검증
필수 확인 케이스:
- 금→토 24h
- 목→화 5일
- 15일
- 20일
- 30일
- 44일
- 50일

### 종료 조건
- 주말 포함 오판정이 제거된다.
- 장기 계산이 4개 필드 기준으로 일관되게 동작한다.

---

## 6. DB 최소 변경 판단

지금 기준으로는 DB를 크게 늘릴 필요가 없다.
우선순위는 아래와 같다.

### 필수
- 장기 4개 컬럼 추가

### 확인 필요
- 운영 페이지에서 파라미터를 어디에 임시 저장할지
- 기존 preview/publish 구조만으로 충분한지

### 보류
- 신규 대형 테이블 구조
- 과도한 프로필/버전 테이블 추가

즉 1차는 **기존 구조 유지 + 장기 4개 컬럼 추가 + 운영 페이지 입력면 확장**으로 보는 게 맞다.
