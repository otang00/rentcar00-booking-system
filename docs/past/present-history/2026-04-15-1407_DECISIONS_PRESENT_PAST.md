# DECISIONS

## 2026-04-15

### 1. 현재 기준점은 `31f55f2` 로 잠근다
- 의미: **DB 연결 전 포인트**
- 이 commit 이후의 검색 DB 연결 작업은 모두 이 기준점 위에서 진행한다.
- 필요 시 여기로 롤백해 다시 출발할 수 있다.

### 2. 다음 목표는 관리자페이지가 아니라 검색 DB 연결이다
- 현재 우선순위는 홈페이지 검색 결과를 우리 DB 기준으로 재현하는 것이다.
- 관리자 페이지, 권한 시스템, 운영자 로그인은 이번 범위에 넣지 않는다.
- 이유: 범위를 과도하게 키우지 않고 핵심 전환부터 끝내기 위해.

### 3. 우리 DB를 직접 읽는 서버 검색 계층을 만든다
- 프론트가 Supabase를 직접 읽지 않는다.
- 프론트는 계속 우리 서버 API만 호출한다.
- 서버 내부에서 `cars + reservations` 를 읽는 검색 서비스 계층을 만든다.
- 이유: 인증/권한/필터/노출 범위를 서버에서 통제하기 위해.

### 4. 기존 검색 endpoint는 가능하면 유지한다
- 프론트 계약을 흔들지 않기 위해, 외부 endpoint는 가능하면 유지한다.
- 실제 교체 대상은 endpoint 자체가 아니라 내부 데이터 source 와 계산 엔진이다.
- 이유: 프론트 변경 범위를 줄이고 롤백을 쉽게 하기 위해.

### 5. 전환 방식은 병행 비교 후 스위치다
- 바로 partner 를 제거하지 않는다.
- 먼저 partner 결과와 DB 결과를 동시에 계산한다.
- diff 를 보고 보정한 뒤 feature flag 로 전환한다.
- 이유: 검색 결과 품질을 수치로 확인하고 안전하게 바꾸기 위해.

### 6. 성공 기준은 "지금과 같은 결과"다
- 감으로 비슷한 수준을 성공으로 보지 않는다.
- 아래 기준을 우선 본다.
  - 결과 차량 수
  - 포함 차량 목록
  - 정렬
  - 가격
  - 제외 사유
- 이유: 전환 후 체감 차이를 최소화하기 위해.

### 7. 기존 present 문서는 history 로 내린다
- 2026-04-15 기준 이전 `present/*` 문서는 `docs/past/present-history/` 로 이동한다.
- 오늘 이후 현행 판단은 새 `present/*` 문서 기준으로 본다.
- 이유: 완료된 IMS sync 기준점과 앞으로의 DB 연결 로드맵을 분리하기 위해.
