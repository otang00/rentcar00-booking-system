## 0. 목적

이 문서는 현시점 구현 대상을 명확히 잠근다.

현재 Phase 1 목표는 아래 하나다.

```plain text
카카오 로그인/카카오싱크는 다음 phase 로 넘기고,
회원정보를 직접 입력해서 가입하는 회원가입 페이지를 먼저 만든다.
```

즉, 지금 만드는 것은:
- 이메일 기반 회원가입 페이지
- 회원정보 입력 폼
- 휴대폰 OTP 인증 포함 가입 흐름
- 주소 입력 포함 가입 흐름
- 필수 약관 동의 포함 가입 흐름

지금 만들지 않는 것은:
- 카카오 로그인
- 카카오싱크 등록/온보딩
- provider identity 연결
- 카카오 자동입력/병합 정책

## 1. 기준점

### 1.1 현재 회원가입 화면 문제
현재 `SignupPage.jsx` 는 아래만 받는다.
- 이메일
- 비밀번호
- 비밀번호 확인

이 상태로는:
- 운영에 필요한 회원 기본정보가 없음
- 휴대폰 인증된 연락처가 없음
- 주소 정보가 없음
- 카카오싱크 증빙용 일반 회원가입 구조도 약함

### 1.2 이번 phase 기준
이번 phase 에서는 회원가입 화면을 아래 구조로 완전히 바꾼다.

```plain text
이름
생년월일
이메일
비밀번호
비밀번호 확인
휴대폰 번호
SMS OTP 인증번호
우편번호
기본주소
상세주소
필수 약관 동의
선택 마케팅 동의
회원가입 버튼
```

## 4. 필드별 확정 규칙

### 4.1 이름
- 필수
- 2자 이상
- trim
- 공백만 입력 불가
- 과도한 특수문자 제한

### 4.2 생년월일
- 필수
- 숫자 8자리
- 예시: `19900101`
- source of truth 로 저장
- 나이는 저장하지 않음

### 4.3 이메일
- 필수
- 이메일 형식 확인
- trim
- 소문자 정규화
- 로그인 ID 로 사용
- 중복 검증 필요

### 4.4 비밀번호
- 필수
- 8자 이상
- 영문 포함
- 숫자 포함
- 공백 불가
- 이메일과 동일값 금지
- 특수문자 권장

### 4.5 비밀번호 확인
- 필수
- 비밀번호와 일치해야 함

### 4.6 휴대폰 번호
- 필수
- 입력 중 자동 하이픈 표시
- 저장은 숫자만
- OTP 성공 전에는 최종 완료 상태 아님

### 4.7 OTP 인증번호
- OTP 발송 후 필수
- 6자리 숫자
- 유효시간 3분 기준
- 실패 5회 제한 기준 권장

### 4.8 우편번호
- 필수
- 주소 검색 결과 입력

### 4.9 기본주소
- 필수
- 주소 검색 결과 입력

### 4.10 상세주소
- 필수
- 직접 입력

### 4.11 약관 동의
필수:
- 서비스 이용약관
- 개인정보 수집 및 이용
- 렌터카 예약 및 대여 조건

선택:
- 마케팅 정보 수신 동의

## 6. 주소 입력 설계

### 6.1 입력 방식
```plain text
우편번호 찾기 클릭
→ 주소 검색 열림
→ 주소 선택
→ 우편번호 자동입력
→ 기본주소 자동입력
→ 상세주소 포커스 이동
```

### 6.2 저장 방식
1차 저장 기준:
- `postal_code`
- `address_main`
- `address_detail`

### 6.3 지금 안 하는 것
- 시/군/구 분리 저장
- 좌표 저장
- 도로명/지번 이중 저장

## 7. OTP 설계

### 7.1 역할
휴대폰 OTP 는 연락처 검증이다.
법적 본인확인이 아니다.

### 7.2 기본 흐름
```plain text
휴대폰 번호 입력
→ 인증번호 받기
→ code_sent
→ 인증번호 입력
→ verify
→ verified
```

### 7.3 UI 기준
- 버튼 기본값: `인증번호 받기`
- 발송 후: `재전송 60초`
- 인증 완료 후: `인증완료`
- 타이머 표시: `03:00`

### 7.4 서버 재검증 원칙
회원가입 제출 시 아래를 서버에서 다시 본다.
- OTP verified 상태
- phone_verified 확정 여부
- 필수 프로필 필드 누락 여부
- 이메일 중복 여부
- 필수 약관 동의 여부

## 8. DB / 저장 계약 초안

### 8.1 `profiles` 현재 확장 기준
현재 phase 기준으로 필요한 컬럼:
- `id`
- `email`
- `name`
- `birth_date`
- `phone`
- `phone_verified`
- `phone_verified_at`
- `postal_code`
- `address_main`
- `address_detail`
- `marketing_agree`
- `profile_status`
- `created_at`
- `updated_at`

### 8.2 저장 원칙
- auth 계정 생성과 profile 저장은 분리된 책임으로 본다.
- 화면에서 받은 회원정보가 profile 최종 기준값이다.
- 빈 profile 자동 생성 흐름은 정상 가입 완료로 보지 않는다.

## 10. 단계별 phase 설계

## Phase 1. 데이터 계약 잠금
### 목적
회원가입 화면이 어떤 필드를 저장하는지 완전히 잠근다.

### 해야 할 일
- `profiles` 최종 컬럼 확정
- `birth_date` 저장 형식 확정
- `phone_verified`/`phone_verified_at` 확정
- `profile_status` 상태값 확정
- 약관 저장 범위 확정

### 종료 조건
- migration 대상 컬럼이 확정된다.
- 프론트 payload 와 서버 저장 필드가 1:1 대응된다.

---

## Phase 2. 화면 계약 잠금
### 목적
회원가입 페이지 동작을 구현 전 단계에서 확정한다.

### 해야 할 일
- 필드 순서 확정
- 섹션 구성 확정
- 에러 문구 확정
- OTP 상태 UI 확정
- 주소 검색 후 포커스 규칙 확정
- 버튼 활성화 조건 확정

### 종료 조건
- 화면 spec 만 보고 구현 가능하다.
- 누락 필드/애매한 흐름이 없다.

---

## Phase 3. API/OTP 계약 잠금
### 목적
회원가입에 필요한 API 호출 순서를 명확히 한다.

### 해야 할 일
- OTP 발송 API 계약 정의
- OTP 검증 API 계약 정의
- signup/profile 저장 순서 정의
- 실패/만료/재전송 규칙 정의

### 종료 조건
- 프론트와 서버 호출 순서가 고정된다.
- OTP 성공 전 가입 완료가 차단되는 구조가 설명 가능하다.

---

## Phase 4. 구현 파일 범위 잠금
### 목적
실제 수정 파일과 책임을 나눈다.

### 수정 대상 후보
- `src/pages/SignupPage.jsx`
- 공용 입력 유틸/컴포넌트
- 주소 검색 연동 파일
- OTP 발송/검증 API
- `server/auth/ensureProfileForUser.js`
- `profiles` migration 파일

### 종료 조건
- 어떤 파일을 왜 수정하는지 설명 가능하다.
- 범위가 과도하게 퍼지지 않는다.

---

## Phase 5. 검증 시나리오 잠금
### 목적
구현 후 무엇으로 통과/실패를 볼지 먼저 정한다.

### 필수 시나리오
- 정상 회원가입 성공
- 이메일 중복 실패
- 비밀번호 조건 미충족 실패
- 비밀번호 불일치 실패
- OTP 미인증 상태 가입 차단
- OTP 만료 후 재요청
- 주소 미입력 가입 차단
- 필수 약관 미동의 가입 차단

### 종료 조건
- 구현 후 테스트 기준이 바로 있다.

---

## 11. 최종 결론

- 지금 phase 는 **카카오를 넘기고 회원정보 입력 회원가입 페이지**를 먼저 만드는 단계다.
- 생년월일과 주소는 이번 phase 에서 실제로 받는다.
- 휴대폰 OTP 인증은 회원가입 완료 조건이다.
- 화면/DB/API/검증을 phase 단위로 먼저 잠근 뒤 구현한다.
- 카카오는 다음 phase 에서 별도 current 기준으로 다룬다.
