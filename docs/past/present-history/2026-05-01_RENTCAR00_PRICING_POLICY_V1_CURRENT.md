# RENTCAR00 pricing policy v1 current

Last updated: 2026-05-01

이 문서는 가격 정책을 복잡한 구조 설명 대신 실제 실행 기준 3개 phase 로 정리한다.
현재 실코드는 이 문서를 아직 완전히 따르지 않을 수 있다.

---

## 1. 목적

이번 정리의 목적은 3개다.

1. 장기 가격 저장 필드를 먼저 확정한다.
2. 운영 페이지에서 조정할 운영파라미터 입력 구조를 만든다.
3. 단기/장기 계산식을 다시 맞춘다.

---

## 2. 전체 원칙

### 2-1. 공식값과 운영값 분리
- DB는 공식 확정 가격 저장소로 본다.
- 운영파라미터는 가격조정 페이지에서 입력/조정하는 값으로 본다.
- 운영파라미터는 가격 생성 기준값이며, 확정 가격 자체와는 구분한다.

### 2-2. 이번 범위에서 고정할 것
- 15일
- 15일 데일리
- 30일
- 30일 데일리
- 일대여 주중/주말 계산 방식
- 장기 계산 구조

### 2-3. 이번 범위에서 조정 가능하게 둘 것
- 기준가격
- 주중 %
- 주말 %
- 월차 할인 관련 값
- 필요 최소 단기 계수

---

## 3. Phase 1. 장기 4개 컬럼 추가

### 목적
장기 계산에 필요한 공식 저장 필드를 DB에 추가한다.

### 추가 대상
- `15일`
- `15일 데일리`
- `30일`
- `30일 데일리`

권장 필드명:
- `d15_total_cost`
- `d15_daily_cost`
- `m1_total_cost`
- `m1_daily_cost`

### 정책 의미
- `d15_total_cost` = 15일 패키지 총액
- `d15_daily_cost` = 15일 초과 구간 계산에 쓰는 일요금
- `m1_total_cost` = 30일 패키지 총액
- `m1_daily_cost` = 30일 초과 잔여일 계산에 쓰는 일요금

### 현재 기준
IMS monthly source 도 동일한 4개 구조를 사용한다.
즉 이번 컬럼 추가는 별도 가정이 아니라 현재 운영 구조와 맞는다.

### 종료 조건
- DB에 장기 4개 필드가 추가된다.
- 어떤 필드가 어떤 계산에 쓰이는지 문서상 모호함이 없다.

---

## 4. Phase 2. 운영 페이지에 운영파라미터 추가

### 목적
운영자가 가격조정 페이지에서 기준값을 정렬해서 입력/조정할 수 있게 한다.

### 운영파라미터 성격
- 운영파라미터는 공식 확정 가격이 아니다.
- 공식 가격을 만들기 위한 입력 기준값이다.
- 필요 시 조정할 수 있다.

### 운영 페이지에서 다룰 핵심 값
- 기준가격
- 주중 %
- 주말 %
- 월차 할인 관련 값
- 필요 최소 단기 계수

### 운영 방식
- 기준가격 기준으로 정렬
- 행 단위 입력/수정
- 미리보기 확인
- 최종 반영 시 공식값 갱신

### 주의
이번 문서 단계에서는 운영파라미터를 복잡한 별도 체계로 확장하지 않는다.
핵심은 운영 페이지에서 입력 가능하게 만드는 것이다.

### 종료 조건
- 운영 페이지에서 기준가격 기준 정렬이 가능하다.
- 운영자가 장기/주중/주말 관련 핵심 파라미터를 직접 조정할 수 있다.

---

## 5. Phase 3. 계산식 재정립

### 목적
단기/장기 계산식을 현재 운영 기대값에 맞게 다시 정리한다.

### 5-1. 일대여 계산
현재 시작요일 기준 계산은 잘못된 케이스가 있다.
목표는 실제 포함된 과금일 기준으로 주중/주말을 분리 계산하는 것이다.

정리:
- 버킷은 총 대여시간 기준 유지
- `billableDays = ceil(totalHours / 24)`
- 각 과금일을 실제 포함된 날짜 기준으로 주중/주말 분리
- 최종 합계:
  - `weekdayDays × weekdayUnitPrice`
  - `weekendDays × weekendUnitPrice`

### 5-2. 장기 계산
장기는 아래 4개 값을 기준으로 계산한다.
- `d15_total_cost`
- `d15_daily_cost`
- `m1_total_cost`
- `m1_daily_cost`

#### 15일 이상 30일 미만
- `min(d15_total_cost + ((days - 15) × d15_daily_cost), m1_total_cost)`

#### 30일 이상
- `months = floor(days / 30)`
- `remain = days % 30`
- `base = months × m1_total_cost`
- `remain < 15` → `min(remain × m1_daily_cost, d15_total_cost)`
- `remain >= 15` → `min(d15_total_cost + ((remain - 15) × m1_daily_cost), m1_total_cost)`
- `total = base + remain_cost`

### 종료 조건
- 일대여 계산 기준이 시작요일 기준에서 실제 포함 과금일 기준으로 바뀐다.
- 장기 계산이 장기 4개 필드 기준으로 일관되게 정리된다.

---

## 6. 지금 문서 기준에서 DB 최소 변경 판단

이번 구조에서 DB 추가는 많이 필요하지 않다.
우선 핵심은 장기 4개 필드 추가다.

즉 현재 기준 최소 변경은:
1. 장기 4개 컬럼 추가
2. 필요 시 운영 페이지 저장용 최소 필드 점검

이 문서 기준으로는 새로운 대형 테이블 구조보다 기존 구조를 유지하면서 필요한 필드만 추가하는 쪽이 우선이다.

---

## 7. 한 줄 결론

**이번 가격 개편은 복잡한 구조 확장보다, 장기 4개 컬럼 추가 → 운영 페이지 운영파라미터 추가 → 계산식 재정립의 3개 phase 로 보는 것이 맞다.**
