## 0. 이번 작업 목표

이번 작업은 아래 2개를 동시에 끝내는 것이다.

1. **비회원 예약접수 시 휴대폰 인증을 필수화한다.**
   - 단, 로그인 회원은 예외로 둔다.
2. **회원가입과 비회원 예약접수 입력값에 말이 안 되는 값이 들어오지 않게 기준을 강화한다.**

이번 작업의 핵심은 단순 UI 추가가 아니라,
**프론트/서버/검증 기준을 같은 규칙으로 잠그는 것**이다.

이번 phase 에서는 특히 아래 UX 규칙까지 같이 잠근다.
- 로그인 회원은 상세 진입 시 운전자 정보가 **잠금 상태**로 들어온다.
- 비회원은 운전자 정보가 **빈 상태**로 들어온다.
- 로그인 회원의 `수정`은 **새 운전자 정보 입력을 시작하는 행위**로 취급한다.
- 비회원의 `인증`은 **예약 접수를 위한 최소 조건 충족 행위**로 취급한다.

## 1. 왜 이 작업을 하는가

### A. 비회원 예약접수 신뢰도 부족
현재는 비회원 예약접수에서 이름/생년월일/전화번호만 맞으면 예약 생성이 가능하다.
즉, 입력자 전화번호가 실제 본인 것인지 확인하는 절차가 없다.

### B. 입력값 품질 기준이 너무 느슨함
현재는 아래 문제가 있다.
- 이름: 사실상 비어있지만 않으면 통과하는 수준
- 생년월일: `30001001` 같은 비현실 날짜도 8자리면 통과 가능
- 전화번호: `09011112222` 같은 휴대폰이 아닌 번호도 길이만 맞으면 일부 경로에서 통과 가능

새 세션이 이 문제를 다시 느슨하게 돌리지 않도록 이번 문서에서 기준을 명확히 잠근다.

## 2. 이번에 잠글 기준

### 2-1. 비회원 예약접수 전화인증 기준
- 회원가입 OTP와 예약 OTP는 **의미가 다르다.**
  - 회원가입 OTP: 계정 연락처 검증 + 회원 상태 승격용
  - 예약 OTP: 현재 상세 화면에서 예약 접수를 진행할 수 있게 하는 **1회성 예약용 토큰**
- 로그인 회원 기본 잠금 상태: **전화인증 불필요**
- 비로그인 사용자: **예약 접수 전 휴대폰 OTP 인증 필수**
- 인증 목적(`purpose`)은 회원가입과 분리한다.
  - 회원가입: `signup`
  - 비회원 예약접수: `guest_booking`
- 예약 OTP는 아래 성격으로 취급한다.
  - 현재 입력된 운전자 전화번호에 바인딩
  - 현재 예약 화면 컨텍스트에 바인딩
  - 짧은 TTL
  - 예약 접수 성공 시 1회 consume
  - `phone_verified` 같은 회원 상태 승격과는 무관
- 예약 생성 API는 비로그인 요청이면 아래를 반드시 확인해야 한다.
  - `phoneVerificationId`
  - `phoneVerificationToken`
  - 인증된 전화번호와 입력 전화번호 일치
  - purpose=`guest_booking`
  - status=`verified`
  - `consumed_at` 없음
  - 만료되지 않음
- 예약 생성 성공 시 guest_booking용 verification 도 `consumed` 처리한다.

### 2-1-0. 예약용 인증 토큰 유효 기준
예약용 인증은 단순 전화번호 인증이 아니라,
**현재 예약 화면에서 예약 접수를 허용하는 1회성 토큰**으로 본다.

토큰은 최소 아래 값에 묶는다.
- `phone`
- `carId`
- `detailToken`
- `deliveryDateTime`
- `returnDateTime`
- `pickupOption`

가능하면 아래도 함께 묶는다.
- `quotedTotalAmount` 또는 `finalAmount`

유효 조건
1. TTL 안에 있어야 함
2. 현재 제출 payload 와 인증 시 컨텍스트가 같아야 함
3. 아직 consume 되지 않았어야 함

권장 TTL
- 예약용 인증은 signup verified token 보다 짧게 둔다.
- 1차 권장값: **10분**

무효화 조건
- 예약 접수 성공
- 전화번호 변경
- 차량 변경
- 일정 변경
- 수령 방식 변경
- 가격 컨텍스트 변경 시 필요하면 재인증

### 2-1-A. 로그인 회원 상세페이지 운전자 정보 규칙
- 로그인 회원이 상세페이지에 들어오면 운전자 정보는 아래 값으로 **자동 프리필 + 잠금 상태**로 보여준다.
  - 이름
  - 생년월일
  - 전화번호
- 잠금 상태에서는 `수정` 버튼만 보인다.
- 이 상태에서 그대로 예약 접수를 누르면 **OTP 없이 진행 가능**하다.

### 2-1-B. 로그인 회원 수정 모드 규칙
- 로그인 회원이 `수정` 버튼을 누르면 모달을 띄운다.
  - 문구: `휴대폰 인증이 필요합니다. 진행하시겠습니까?`
- 사용자가 확인을 누르면 아래를 동시에 수행한다.
  1. 기존 프리필 값을 비운다.
  2. 운전자 정보 입력을 편집 가능 상태로 전환한다.
  3. `수정` 버튼을 `전화번호 인증` 버튼으로 바꾼다.
- 수정 모드에서 인증 완료 전에는 예약 접수가 불가하다.
- 인증 완료 후에는 현재 입력값 기준으로 다시 **잠금 상태**로 돌린다.
- 다시 수정하려면 다시 `수정` → 모달 확인 → 재인증 절차를 거친다.

즉 로그인 회원 흐름은 아래처럼 잠근다.
- 잠금 상태 → `수정`
- 모달 확인
- 입력값 비움 + 편집 상태
- `전화번호 인증`
- 인증 완료
- 다시 잠금 상태

### 2-1-C. 비회원 상세페이지 운전자 정보 규칙
- 비회원이 상세페이지에 들어오면 운전자 정보는 **빈 상태**다.
- 비회원에게는 `수정` 버튼을 두지 않는다.
- 운전자 정보 입력폼은 기본 노출
- `수정` 버튼은 숨김 또는 비활성화
- CTA는 `전화번호 인증` 버튼으로 둔다.
- 인증 완료 후에는 현재 입력값 기준으로 다시 잠금 상태로 돌린다.
- 다시 수정하려면 입력 잠금을 풀고 재인증부터 다시 진행한다.

즉,
- 회원: `수정` → 필요 시 `전화번호 인증`
- 비회원: 처음부터 `전화번호 인증`

### 2-1-D. 버튼/상태 설계 기준
- 로그인 회원 잠금 상태 버튼: `수정`
- 로그인 회원 수정 상태 버튼: `전화번호 인증`
- 비회원 버튼: `전화번호 인증`
- 예약 가능 상태는 아래처럼 분리한다.
  - 회원 잠금 상태: 즉시 예약 가능
  - 회원 수정 상태: 인증 전 예약 불가
  - 비회원: 인증 전 예약 불가
- `예약 접수` 버튼은 별도로 유지한다.
- 인증이 안 된 상태에서 `예약 접수`를 누르면
  - 제출은 막고
  - `전화번호 인증을 완료해 주세요.` 경고를 띄운다.

### 2-1-E. 서버 판단 기준
서버는 단순히 로그인 여부만 보면 안 된다.

- 로그인 회원 + 잠금 기본값 그대로 제출
  - OTP 없이 예약 허용 가능
- 로그인 회원 + 수정 상태 제출
  - 예약용 인증 토큰 검증 필요
- 비회원 제출
  - guest booking OTP 필수

이번 phase 단순화 권장안:
- 로그인 회원 수정 상태와 비회원 예약접수는 서버에서 같은 `예약용 인증` 검증 축으로 본다.
- 다만 signup OTP 와는 명확히 분리한다.

### 2-2. 이름 검증 기준
이름은 아래 기준으로 잠근다.
- trim 후 길이 **2자 이상 5자 미만**
- 한글, 영문, 공백 허용
- 숫자/기호/이모지 금지
- `홍길동님`, `test123`, `@@@`, `abcdef` 같은 값은 막는 쪽으로 간다.
- 사용자 메시지는 너무 공격적으로 하지 말고 `이름 형식을 확인해 주세요.` 수준으로 통일한다.

예시
- 허용: `홍길동`, `JIN`, `JIN WOO`
- 차단: `홍길동님`, `test123`, `@@@`, `abcdef`

### 2-3. 생년월일 검증 기준
생년월일은 아래 기준으로 잠근다.
- 8자리 숫자 필수
- 실제 달력 날짜여야 함
- 미래 날짜 금지
- 비현실 연도 제한 필요
- 1차 권장 기준:
  - 연도 하한: `1900-01-01`
  - 오늘 날짜 이하만 허용
- 즉 아래는 실패해야 한다.
  - `30001001`
  - `19990231`
  - `20261301`
  - `00000000`

### 2-4. 휴대폰 번호 검증 기준
휴대폰 번호는 아래 기준으로 잠근다.
- normalize 후 `^01\d{8,9}$`
- 즉 한국 휴대폰 번호만 허용
- `090`, `080`, 지역번호, 유선번호는 실패
- 회원가입 / OTP / 비회원 예약접수 / 비회원 조회가 모두 같은 규칙을 써야 한다.

## 3. 기준점

### 현재 확인된 사실
- 회원가입 OTP는 이미 `purpose='signup'` 기준으로 구현되어 있다.
- OTP API는 현재 signup 외 purpose 를 거부한다.
- 비회원 예약접수는 아직 OTP 없이 생성 가능하다.
- 회원 상세 진입 프리필은 이미 들어가 있다.
- 회원 상세 진입 시 입력 잠금 / 수정 버튼 / 예약용 OTP 분기 UI는 아직 구현 안 되어 있다.
- 예약 생성 직후 상태는 `confirmation_pending + pending` 이다.

- `api/auth/otp/[action].js`
- `server/auth/phoneOtp.js`

## 4. Phase 계획

### Phase 1. 공통 입력 검증 규칙 함수 잠금
목적
- 이름 / 생년월일 / 휴대폰 번호 규칙을 프론트/서버가 다른 방식으로 판단하지 않게 한다.

수정 범위
- 공통 validation 유틸 신설 또는 기존 normalize/validate 유틸 확장
- 후보 파일:
  - `server/booking-core/bookingIdentity.js`
  - `src/services/reservationForm.js`
  - 필요 시 공용 util 신규 파일

종료 조건
- 이름/생년월일/휴대폰 번호 규칙이 코드상 한 군데 또는 명확히 대응되는 구조로 잠긴다.
- 회원가입/비회원 예약접수/비회원 조회가 같은 기준을 사용한다.

### Phase 2. 회원가입 검증 강화
목적
- signup 화면과 signup API가 같은 기준으로 입력을 거른다.

수정 범위
- `src/pages/SignupPage.jsx`
- `api/auth/[action].js`
- 필요 시 OTP 요청 전 휴대폰 번호 검증 메시지 정리

종료 조건
- 이름 5자 이상, 비현실 생년월일, 휴대폰이 아닌 번호가 UI와 서버 모두에서 차단된다.
- 프론트/서버 에러 메시지가 충돌하지 않는다.

### Phase 3. 비회원 예약접수 OTP 도입
목적
- 비로그인 사용자의 예약 접수에 전화번호 실소유 검증을 추가한다.

수정 범위
- `api/auth/otp/[action].js`
  - `guest_booking` purpose 허용
- `src/components/CarDetailSection.jsx`
  - 비로그인일 때만 OTP UI 노출
  - 로그인 회원은 기존 프리필/수정 가능 유지
- `api/guest-bookings/[action].js`
  - guest_booking verification 재검증 + consumed 처리

종료 조건
- 로그인 회원은 OTP 없이 예약 가능
- 비로그인 사용자는 OTP 없이는 예약 생성 불가
- 인증된 번호와 입력 번호가 다르면 예약 생성 불가
- 인증 토큰은 현재 예약 화면 컨텍스트와 불일치하면 재사용 불가

### Phase 3-A. 회원/비회원 상세페이지 상태 머신 적용
목적
- 상세페이지에서 회원/비회원/수정상태/인증상태를 명확히 분리한다.

상태 정의
1. 회원 잠금 상태
   - 프로필값 프리필
   - 입력 잠금
   - `수정` 버튼 노출
   - OTP 없이 예약 가능
2. 회원 수정 상태
   - 모달 확인 후 진입
   - 입력값 비움
   - 입력 편집 가능
   - 예약용 인증 필요
   - 인증 전 예약 불가
3. 비회원 기본 상태
   - 입력값 비어 있음
   - 입력 편집 가능
   - `수정` 버튼 없음
   - `전화번호 인증` 버튼 노출
   - 인증 전 예약 불가
4. 인증 완료 상태
   - 현재 입력 전화번호 기준 인증 완료
   - 현재 예약 화면 컨텍스트 기준 인증 완료
   - 다시 잠금 상태
   - 예약 접수 가능

## 5. 이번 작업에서 특히 조심할 것

### A. signup OTP와 guest booking OTP를 섞지 말 것
- purpose 를 분리하지 않으면 회원가입용 인증값으로 비회원 예약을 통과시키는 꼬임이 생길 수 있다.

### A-1. 예약 OTP를 회원 상태 승격처럼 다루지 말 것
- 예약 OTP는 `phone_verified` 로 저장되면 안 된다.
- 예약 접수 화면을 통과시키는 최소 권한 토큰으로만 봐야 한다.

### A-2. 예약용 인증은 화면 컨텍스트와 묶을 것
- 인증만 받아두고 다른 차량/다른 일정/다른 가격 컨텍스트에 재사용되면 안 된다.
- 서버 submit 단계에서 인증 토큰의 context hash 를 다시 검증해야 한다.

### B. 프론트만 막고 서버를 열어두지 말 것
- 이번 작업은 반드시 서버 최종 검증까지 같이 잠가야 한다.

### C. 로그인 회원 예외 처리 명확화
- 로그인 상태라고 무조건 OTP 면제면 안 된다.
- 로그인 회원은 **잠금 기본값 제출일 때만 면제**다.
- `수정` 모달을 확인해 편집 상태로 전환한 순간부터 예약용 인증 흐름이 필요하다.

### D. 비회원 조회 기준과 예약접수 기준 분리 여부
- 이번 요청은 “비회원 예약접수”와 “회원가입” 기준 강화가 핵심이다.
- 하지만 lookup/cancel 쪽 검증도 같은 규칙을 쓰게 맞추는 편이 장기적으로 안전하다.

## 6. 실행 전 마지막 확인 필요 항목
1. 예약용 인증 context hash 최종 구성값
   - `phone`
   - `carId`
   - `detailToken`
   - `deliveryDateTime`
   - `returnDateTime`
   - `pickupOption`
   - 필요 시 가격값
2. 예약용 인증 TTL 최종값
   - 권장 10분
3. 인증 완료 후 잠금 상태 UI 표현
   - 인증 완료 배지/문구를 둘지 여부

이 3개는 실행 직전에 짧게 잠그고 들어간다.

## 7. 한 줄 결론
이번 작업은
- **비로그인 예약접수에 OTP를 붙이고**
- **회원가입/비회원 예약 입력값 검증을 현실적인 수준으로 강화하는 것**이며,
핵심은 **프론트와 서버가 같은 기준을 쓰게 잠그는 것**이다.
