# RENTCAR00 PRICING_HUB

## 역할
이 문서는 RENTCAR00의 장기 가격 허브 기준서다.
`PRICING_HUB`는 IMS, 찜카, 자사플랫폼 3개 채널 가격을 조율하고, 공통 가격 모델을 기준으로 채널별 산출/배포를 관리하는 허브를 뜻한다.
이 문서는 실행 중인 current 문서가 아니라 장기 구조와 정책을 잠그는 정책 문서다.

---

## 1. 허브의 목적
### 1-1. 왜 허브가 필요한가
현재 가격 관련 구조는 크게 2축으로 나뉜다.
1. 자사플랫폼 메인 검색/상세/예약 가격 계산
2. 외부/운영 채널별 가격 입력, 조정, 배포

장기적으로는 아래 3개 채널 가격을 별도 감으로 관리하지 않고, 하나의 공통 기준에서 조율해야 한다.
- IMS
- 찜카
- 자사플랫폼

`PRICING_HUB`의 목적은 아래다.
- 공통 가격 모델을 하나로 관리한다.
- 채널별 입력/출력 규칙을 분리한다.
- preview 없이 직접 배포하지 않는다.
- 어떤 값이 원본(source of truth)이고 어떤 값이 파생값인지 헷갈리지 않게 만든다.

### 1-2. 이번 current 작업과의 관계
이번 `docs/present/*CURRENT.md` 가격 작업은 장기 허브 완성이 아니라,
**자사플랫폼 메인 가격 계산식 정렬**이 우선이다.
즉 current 작업은 허브의 장기 설계와 분리해서 본다.

---

## 2. 채널 정의
### 2-1. IMS
- 장기적으로는 운영 기준 가격/상품 정책의 한 축이다.
- 월요금/보름요금 등 별도 구조가 있을 수 있다.
- 허브는 IMS 입력 구조를 그대로 복제하는 것이 아니라, 필요한 기준값을 추출/매핑하는 역할을 한다.

### 2-2. 찜카
- 찜카는 실제 입력용 배수/템플릿 산출 기준이 존재한다.
- 현재 확인된 `tmp/zzimcar-fee-section-2111-target-template.csv` 류 산출물은 자사플랫폼 current 작업에서도 참고 기준이 된다.
- 장기 허브에서는 찜카용 출력 포맷/배수/반올림 규칙을 별도 채널 정책으로 본다.

### 2-3. 자사플랫폼
- 현재 `rentcar00-booking-system` 메인 검색/상세/예약 흐름이 대상이다.
- 현재 active 구현 기준은 legacy 가격 경로와 `calculateGroupPrice.js` 기반이다.
- 허브는 장기적으로 자사플랫폼 가격도 공통 모델에서 산출되도록 가야 하지만, 지금은 과도기 상태일 수 있다.

---

## 3. 공통 가격 모델 원칙
### 3-1. 공통 모델이 먼저다
허브는 채널별 화면/API를 먼저 만드는 구조가 아니다.
먼저 아래 공통 가격 모델을 잠그고, 채널별 출력은 그 다음에 파생한다.

공통 모델 예시 축:
- 기준 24시간 금액
- 주중/주말 비율 또는 금액
- 단기 구간 규칙
- 중기 구간 규칙
- 장기 구간 규칙
- 반올림 규칙
- 예외 차종/등급 규칙

### 3-2. source of truth 구분
허브에서 항상 구분해야 하는 것은 아래다.
1. 원본 입력값
2. 허브 내부 계산값
3. 채널별 출력값

이 3개가 섞이면 안 된다.

예시:
- `base24h` 는 원본 입력값일 수 있다.
- `weekdayDaily`, `weekendDaily` 는 허브 내부 계산값일 수 있다.
- IMS용 월요금/찜카용 배수표/자사플랫폼 검색 노출가는 채널별 출력값일 수 있다.

### 3-3. 채널별 override 허용
공통 모델이 있어도 채널별 제약은 남는다.
따라서 허브는 아래를 허용할 수 있어야 한다.
- 채널별 라운딩 규칙
- 채널별 최소/최대 캡
- 채널별 장기 구간 해석
- 채널별 비노출/예외 처리

단, override 는 공통 모델을 무너뜨리는 임시 땜질이 아니라,
**명시된 채널 정책**으로 관리해야 한다.

---

## 4. 허브의 입력 / 계산 / 출력 구조
### 4-1. 입력
허브 입력은 운영자가 직접 확정하는 값이다.
예시:
- 기준 24h
- 주중/주말 기준
- 7일/14일/30일 앵커
- 차종 등급
- 장기 특별 규칙

### 4-2. 계산
허브 계산은 입력값을 공통 모델로 정리하는 단계다.
예시:
- 단기 버킷 산출
- 장기 앵커/증분 계산
- 채널별 변환 전 preview 생성

### 4-3. 출력
출력은 채널별 publish/sync 대상이다.
예시:
- 자사플랫폼 가격 계산 기준
- 찜카 입력 템플릿
- IMS 반영용 구조

출력 단계는 preview 와 publish 를 분리한다.

---

## 5. preview / publish / sync 원칙
### 5-1. preview 선행
허브는 저장 즉시 외부 반영 구조보다, preview 확인 후 publish/sync 하는 단계형 구조가 맞다.

최소 단계:
1. 입력 확정
2. preview 생성
3. 채널별 비교
4. publish/sync 실행
5. 결과 기록

### 5-2. 채널별 배포는 분리
아래는 한 번에 묶어버리면 안 된다.
- 자사플랫폼 계산 반영
- 찜카 배포
- IMS 반영

채널별 실패/재시도/검증 방식이 다를 수 있으므로 분리 관리가 원칙이다.

### 5-3. 실행 로그 필요
장기 허브에서는 아래 기록이 필요하다.
- 어떤 입력값으로 계산했는지
- 어떤 채널에 언제 배포했는지
- preview 와 실제 publish 결과가 달랐는지
- 실패 채널이 어디인지

---

## 6. 현재 booking-system 안 구현체와의 관계
현재 확인된 `pricing_hub` 구현 흔적은 `rentcar00-booking-system` 내부에 있다.
예시:
- `api/admin/pricing-hub.js`
- `src/pages/AdminPricingHubPage.jsx`
- `pricing_hub_periods`
- `pricing_hub_rates`
- `pricing_hub_overrides`
- `pricing_hub_previews`

현재 판단:
1. 이것은 장기 허브의 초기 구현체 또는 실험적 운영면일 수 있다.
2. 하지만 아직 장기 `IMS + 찜카 + 자사플랫폼` 허브 전체 구조가 완성된 것으로 보지는 않는다.
3. 따라서 현재 booking-system 안의 pricing_hub 구현과 장기 허브 정책 문서를 동일시하지 않는다.

## 7. 문서 역할 분리
### 이 문서가 다루는 것
- 장기 허브 목적
- 채널 구조
- 공통 가격 모델
- publish/sync 원칙
- booking-system 구현과의 관계

### 이 문서가 다루지 않는 것
- 특정 날짜 current 작업 진행상황
- 이번 자사플랫폼 계산식 상세 수식
- 임시 phase 체크리스트
- 단일 배포 실행 메모

그런 내용은 `present/*CURRENT.md` 또는 `past/` 로 분리한다.
