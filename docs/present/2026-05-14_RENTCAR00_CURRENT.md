## 비회원 예약조회 구조 전환 메모
### 현재 문제
- 현재 비회원 예약조회는 이름 + 휴대폰 + 생년월일 정확 일치 기반이다.
- 조회 성공 후 별도 lookup session/token 이 없어서, 개인정보를 아는 사람은 반복 조회 시도가 가능하다.
- 외국인 이름/영문 이름은 대소문자/표기 차이 이슈도 생긴다.

### 전환 목표
- 비회원 예약조회는 개인정보 지식 기반이 아니라 **휴대폰 점유 인증 기반**으로 바꾼다.
- 인증 성공 후 **짧은 만료시간 lookup token** 을 발급한다.
- 토큰 유효시간 동안만 예약목록/상세/취소를 허용한다.
- 만료되면 다시 비회원 예약조회 인증 화면으로 보낸다.
- 단, **회원 계정이 있는 휴대폰 번호는 비회원 흐름 자체를 차단**한다.

### 목표 UX
1. `/guest-bookings` 진입
2. 휴대폰 번호 입력
3. OTP 인증 성공
4. 서버가 lookup token 발급
5. 프론트는 예약목록 화면으로 이동
6. 화면 상단에 `인증 유효시간` 표시
7. 토큰 유효시간 동안 목록 새로고침 / 취소 가능
8. 토큰 만료 시 다시 인증 화면으로 복귀

### lookup token 방향
- 목적: 비회원 예약조회 전용 임시 접근권한
- 만료시간: 짧게 유지 (예: 10분)
- 권한 범위:
  - 같은 휴대폰 번호 기준 예약목록 조회
  - 같은 인증 상태에서 예약취소
- 저장 방향:
  - 서버 검증 가능한 서명 토큰 또는 DB 검증 토큰 중 하나로 구현 검토
- UI:
  - 상단 배지/문구로 남은 인증시간 표시
  - 만료 직전 재인증 유도 문구 검토

### 보안 기준
- 이름/생년월일은 조회 권한의 핵심 조건에서 제거한다.
- 휴대폰 OTP 없이는 조회/취소 불가로 본다.
- lookup token 만료 후에는 재조회/취소 모두 재인증이 필요하다.
- 기존 guest lookup protection(rate limit / lock / delay)는 유지한다.
- 모든 OTP 발송 전에는 **회원 전화번호 존재 여부를 먼저 확인**한다.
- 회원 전화번호면 OTP 발송 대신 **로그인 유도 문구**를 먼저 노출하고 흐름을 차단한다.

## 회원 전화번호 사전 차단 기준
### 적용 지점
1. 회원가입 OTP 발송 전
2. 비회원 예약 OTP 발송 전
3. 비회원 예약조회 OTP 발송 전
4. 비회원 결제 진행 API 진입 시 최종 재확인

### 차단 기준
- 해당 휴대폰 번호로 회원 계정이 존재하면 비회원 흐름 진입을 차단한다.
- OTP는 발송하지 않는다.
- 안내 문구는 로그인 유도형으로 통일한다.

### 기본 안내 문구
- 공통: `이미 가입된 휴대폰 번호입니다. 로그인 후 진행해 주세요.`
- 예약조회: `회원 예약은 로그인 후 예약내역에서 확인해 주세요.`

## 비회원 예약 → 회원 예약 편입 기준
### 대상
- `user_id` 가 없는 비회원 예약
- 회원가입 완료한 계정의 휴대폰 번호와 일치하는 예약

### 제외 대상
- 반납시간이 현재시간보다 이전인 예약
- 이미 배차/반납이 끝난 예약
- 취소 완료 / 종료 완료 성격의 예약

### 편입 방향
- 회원가입 완료 직후, 아직 끝나지 않은 비회원 예약을 회원 예약으로 편입한다.
- 편입 후에는 비회원 예약조회에서 제외하고 회원 예약내역에서 보이게 한다.
- 비회원 결제 진행 시점에도 동일 기준으로 한 번 더 재확인한다.

### 편입 안내 문구
- 회원가입 화면/완료 직전 또는 완료 안내에 아래 취지 문구를 둔다.
- `진행 중인 비회원 예약이 있으면 회원 예약내역으로 자동 이전됩니다.`
- 필요하면 보조 문구로 `이미 종료된 예약은 이전 대상에서 제외됩니다.` 를 함께 노출한다.

## 비회원 예약조회 전환 실행 phase
### 목적
- 비회원 예약조회/취소를 휴대폰 OTP 인증 + 짧은 lookup token 기준으로 재구성한다.

### 기준점
- 현재는 이름 + 휴대폰 + 생년월일 기반 direct lookup 이다.
- 앞으로는 휴대폰 OTP 인증 성공 여부가 유일한 조회 권한 기준이다.

### Phase 1. 인증/토큰 기준 잠금
- lookup token payload / 만료시간 / 허용 액션 범위를 잠근다.
- OTP 인증 완료 후 어떤 응답을 줄지 API contract 를 잠근다.
- 종료조건: 인증 성공 후 목록/취소 허용 기준을 문장으로 명확히 설명할 수 있다.

### Phase 2. 서버 조회/취소 구조 전환
- guest lookup / cancel 입력값을 이름/생년월일 의존에서 분리한다.
- lookup token 검증 로직을 추가한다.
- 휴대폰 기준 예약목록 조회/취소 구조로 바꾼다.
- 종료조건: 서버가 유효 token 없이는 목록/취소를 허용하지 않는다.

### Phase 3. 프론트 조회 화면 전환
- `GuestBookingsPage` 를 인증 화면 + 예약목록 화면 2단 구조로 바꾼다.
- OTP 인증 성공 시 목록 화면으로 전환한다.
- 상단에 남은 인증시간을 노출한다.
- 종료조건: 토큰 상태에 따라 인증 화면/목록 화면이 명확히 갈린다.

### Phase 4. 만료/에러/재인증 처리
- 토큰 만료 시 재인증 화면 복귀
- 만료 임박 안내 문구 검토
- 취소 시도 중 만료된 경우 처리 방식 통일
- 종료조건: 만료/실패/재시도 UX 가 흔들리지 않는다.

### Phase 5. 검증
- OTP 인증 성공 → 목록 조회
- 토큰 유효시간 내 새로고침 / 재조회
- 토큰 만료 후 재인증 요구
- 취소 성공 / 취소 만료 처리
- build 검증
- 종료조건: 보안 기준과 UX 기준이 모두 충족된다.

- 편입 후 회원 예약내역에서 조회 가능하도록 기존 조회 구조와 충돌 없는지 점검

#### 종료조건
- 편입 대상 예약만 회원 계정으로 연결된다.
- 종료된 예약은 편입되지 않는다.

#### 리스크
- 상태 정의가 흩어져 있으면 종료 예약 제외 조건이 빠질 수 있음.
- 회원 예약내역 조회 조건과 편입 후 상태 노출 경로를 같이 확인해야 함.

### Phase 5. 회원가입 완료 흐름 연결
#### 작업
- 회원가입 성공 직후 편입 서비스 실행
- 편입 성공/편입 없음 모두 회원가입 자체는 정상 완료 처리
- 필요 시 실패 처리 정책을 잠근다.

#### 종료조건
- 회원가입 직후 진행 중 비회원 예약이 자동으로 회원 예약으로 연결된다.
- 편입 대상이 없으면 조용히 통과한다.

#### 검증 포인트
- 비회원 예약 없는 신규 가입자
- 진행 중 비회원 예약 있는 가입자
- 종료된 비회원 예약만 있는 가입자

### Phase 6. 회원가입 안내 문구 반영
#### 작업
- 회원가입 화면 또는 완료 직전 안내 영역에 자동 이전 정책 문구 추가
- 회원 번호 차단 문구와 자동 이전 안내 문구의 톤을 맞춘다.

#### 종료조건
- 사용자가 가입 시점에 예약 이전 정책을 읽을 수 있다.

### Phase 7. 통합 검증
#### 검증 항목
- 회원 번호 `signup` OTP 차단
- 회원 번호 `guest_booking` OTP 차단
- 회원 번호 `guest_lookup` OTP 차단
- 비회원 결제 우회 차단
- 회원가입 직후 진행 중 비회원 예약 편입
- 종료 예약 편입 제외
- build 검증

#### 종료조건
- 차단 / 편입 / 안내가 모두 일관되게 동작한다.
- 다음 단계(커밋/배포)로 넘길 수 있는 상태가 된다.
