# 02. BUILD FLOW

## 목적
이 문서는 `premove-clone` 제작 순서와 각 단계의 진입 조건, 완료 기준을 정리한다.

---

## 전체 순서
1. 프로젝트 기준선 정리
2. 프론트 데이터 구조 전환
3. partner 프록시 조회 구현
4. 차량 상세/예약 페이지 상태화
5. 예약 생성/결제 준비 흐름 구현
6. 실제 결제/IMS 연동
7. 안정화 및 운영 대응

---

## STEP 0. 기준선 정리
### 목표
프로토타입 문서를 정리하고 최종 기준 문서를 확립한다.

### 해야 할 일
- 중복/구버전 문서 archive 이동
- 최종 목표 문서 정리
- 메인 프롬프트 문서 정리
- 제작 플로우 문서 정리
- phase 문서 틀 생성

### 완료 기준
- 앞으로 참고할 문서가 한 방향으로 정리되어 있다.

---

## STEP 1. 프론트 데이터 구조 전환
### 목표
더미 UI를 API 기반 UI 구조로 바꾼다.

### 해야 할 일
- 컴포넌트에서 mock 직접 참조 제거
- service/api layer 추가
- URL 기반 검색 상태 정의
- 메인 페이지 로딩/에러/빈 결과 상태 추가

### 선행 조건
- 문서 정리 완료

### 완료 기준
- 메인/상세 페이지가 mock 직결 구조에서 벗어난다.

---

## STEP 2. partner 프록시 조회 구현
### 목표
실제 차량 목록을 받아오게 만든다.

### 해야 할 일
- partner URL builder 작성
- partner 응답 fetch
- `_rsc` 파싱
- 차량/업체 DTO 매핑
- `/api/search-cars` 구현

### 완료 기준
- 검색 조건 변경 시 실제 차량 목록이 내려온다.

---

## STEP 3. 차량 상세/예약 페이지 상태화
### 목표
정적인 상세 화면을 예약 준비 화면으로 바꾼다.

### 해야 할 일
- 목록→상세 검색 상태 전달
- 선택 차량 데이터 주입
- 예약자 입력 폼 구성
- 약관 동의 상태 구성
- 결제 전 검증 로직 구성

### 완료 기준
- 상세 페이지가 실제 사용자 입력을 받을 수 있다.

---

## STEP 4. 예약 생성/결제 준비 흐름 구현
### 목표
결제 직전까지 서버와 맞물리는 예약 흐름을 만든다.

### 해야 할 일
- 예약 payload 정의
- `/api/reservations/prepare` 설계
- 입력값 검증
- 완료 화면 초안 구현

### 완료 기준
- 예약 버튼 이후 흐름을 mock이 아니라 실제 구조로 시뮬레이션 가능

---

## STEP 5. 실제 결제/IMS 연동
### 목표
실예약 처리

### 해야 할 일
- PG 연동
- 결제 성공 확인
- IMS 예약 생성
- DB 저장
- 실패 보상 정책 구현

### 완료 기준
- 실제 예약 생성 가능

---

## STEP 6. 안정화
### 목표
운영 가능한 품질 확보

### 해야 할 일
- partner 포맷 변경 대응성 점검
- 에러 로깅/재시도
- 빈 결과/오류 UX 개선
- 문서/결정 로그 정리

### 완료 기준
- 운영 중 예외 상황에 대응 가능

---

## 지금 가장 먼저 해야 할 단계
현재 시점의 우선순위는 아래다.
1. STEP 1
2. STEP 2
3. STEP 3

즉,
**더미 구조 청산 → partner 프록시 조회 → 예약 페이지 상태화**
이 순서가 가장 중요하다.
