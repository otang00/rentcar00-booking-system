# 01. MAIN PROMPT

## 목적
이 문서는 앞으로 `premove-clone` 작업을 수행하는 코딩 에이전트/워커에게 주는 **최상위 작업 기준 프롬프트**다.

---

## 역할
너는 `premove-clone`을 단순 디자인 클론이 아니라,
**partner.premove.co.kr 기반 차량 조회를 우리 UX로 재구성하는 예약 서비스 프론트/서버 구현자**다.

---

## 제품 목표
- 메인 페이지에서 검색 조건 입력
- 검색 조건 기반 차량 목록 조회
- 차량 클릭 시 차량 예약 페이지 이동
- 예약자 입력/약관 동의/결제 진입
- 이후 결제 성공 시 예약 저장 및 IMS 연동 가능 구조 확보

---

## 현재 기술 전략

### 차량 조회
- partner URL 결과를 우리 서버가 호출한다.
- 응답(`_rsc`/서버 렌더 결과)을 파싱한다.
- 필요한 차량/업체 정보만 JSON으로 가공한다.
- 프론트는 우리 API만 호출한다.

### 예약/결제
- 조회와 예약 생성은 분리된 단계로 다룬다.
- 목록은 partner 파싱 기반으로 시작하되,
- 예약 생성은 우리 서버 플로우로 설계한다.

---

## 절대 규칙

### 1. 프론트에서 외부 의존 직접 호출 금지
- partner 직접 fetch 금지
- IMS 직접 fetch 금지
- 프론트는 내부 API만 사용

### 2. 메인=검색 시작점
- 메인 페이지는 랜딩이 아니라 검색 페이지다.
- 검색 상태는 URL과 동기화되어야 한다.

### 3. 상세 페이지는 예약 준비 화면이어야 한다
- 차량 정보 표시만으로 끝나면 안 된다.
- 예약 입력 흐름이 있어야 한다.

### 4. 더미 데이터 직접 참조 금지
- 컴포넌트가 mock 파일을 직접 import 하지 않게 한다.
- 데이터는 service/API 계층을 통해 공급한다.

### 5. 교체 가능한 구조 유지
- partner 파싱 로직은 서버 내부 service layer에 격리한다.
- 나중에 IMS 직접 연동으로 교체 가능해야 한다.

### 6. UI보다 흐름 우선
- 예쁜 정적 화면보다 상태와 데이터 흐름을 먼저 맞춘다.

---

## 구현 원칙

### 프론트 원칙
- URL 쿼리 = 검색 상태의 기준
- 페이지는 service/api를 통해서만 데이터 수신
- 로딩/에러/빈 결과 상태 명확히 표시
- 상세 페이지는 검색 조건을 유지한 채 이동

### 서버 원칙
- partner URL builder 분리
- fetcher 분리
- parser 분리
- DTO mapper 분리
- 프론트에 외부 응답 원문을 그대로 전달하지 않음

### 문서 원칙
- 변경 방향이 생기면 문서를 먼저 갱신
- 중복 설계 금지
- 단계가 바뀌면 phase 문서 기준으로 움직임

---

## 우선순위 원칙
1. 데이터 흐름
2. URL/라우팅 일관성
3. 상태 검증
4. 예약 준비 화면 완성도
5. 시각 polish

---

## 금지 사항
- 관리자 기능 확장
- 불필요한 상태관리 라이브러리 추가
- 추측성 API 설계 남발
- partner 응답 형식을 프론트 전역 타입으로 직접 확산
- 페이지별 중복 로직 복붙

---

## 최종 판단 기준
좋은 구현은 아래와 같다.
- 메인 검색 → 목록 → 상세 → 예약 준비가 하나의 흐름으로 이어진다.
- 외부 데이터 소스가 바뀌어도 프론트 수정 범위가 작다.
- 지금 당장 partner 파싱으로 돌릴 수 있다.
- 나중에 결제/IMS를 붙일 수 있다.
