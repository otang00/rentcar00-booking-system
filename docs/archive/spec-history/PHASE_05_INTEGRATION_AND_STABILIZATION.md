# PHASE 05. INTEGRATION AND STABILIZATION

## 목적
실제 결제, IMS 예약 생성, 저장, 예외 처리까지 포함한 운영 가능한 구조를 완성한다.

---

## 이 단계에서 해결할 문제
- 결제 성공 이후 실제 예약 생성 순서 확정 필요
- IMS 생성/DB 저장/완료 노출 사이의 실패 분기 처리 필요
- partner 기반 조회 구조와 실제 예약 생성 구조를 분리해서 운영해야 함

---

## 선행 기준
- PHASE 04 완료
- `prepareId` 또는 동등한 예약 준비 개념 확보
- 결제사/IMS 연결 전략 결정

---

## 핵심 흐름
1. 예약 준비 성공
2. 결제 시도
3. 결제 성공 확인
4. IMS 예약 생성
5. 우리 DB 저장
6. 완료 페이지 노출

---

## 실패 시나리오 상세

### A. 결제 성공 + IMS 실패
대응 필요
- 관리자 확인 필요 상태 저장
- 자동 환불 여부 정책
- 사용자 안내 문구

### B. IMS 성공 + DB 저장 실패
대응 필요
- 재시도 큐 또는 보정 스크립트
- 중복 저장 방지 key

### C. partner 조회 데이터와 결제 시점 조건 불일치
대응 필요
- 결제 직전 재검증
- 가격/가능 여부 차이 발생 시 차단

---

## 운영 보강 항목
- 에러 로깅
- 예약 단계별 상태 추적
- parser 실패 모니터링
- partner 포맷 변경 감지 포인트
- 고객센터 대응 문구 정리

---

## 기술 원칙
- 목록 조회 소스와 예약 생성 소스를 동일하다고 가정하지 않는다.
- partner 파싱은 조회 전략이다.
- 실제 예약 생성은 더 안정적인 서버 흐름으로 분리한다.
- 외부 실패에 대비한 idempotency / retry 전략이 필요하다.

---

## 검증 포인트
- [ ] 결제 성공 이후 처리 순서가 문서화되어 있는가
- [ ] IMS 실패 시 보상 정책이 있는가
- [ ] DB 저장 실패 시 재처리 전략이 있는가
- [ ] 완료 페이지 노출 기준이 명확한가
- [ ] 운영 로그로 상태 추적이 가능한가

---

## 완료 기준
- 결제 후 실제 예약 생성이 가능하다.
- 실패 상황에서 손실/중복 위험을 통제한다.
- 운영 대응 포인트와 예외 흐름이 정리되어 있다.
