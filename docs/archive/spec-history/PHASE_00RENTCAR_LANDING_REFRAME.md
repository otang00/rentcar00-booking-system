# PHASE. 00RENTCAR LANDING REFRAME

## 목적
`00rentcar.com`의 **홈페이지 셸 감성**만 선택적으로 차용해서,
현재 `premove-clone` 위에 **브랜딩 랜딩 + 예약 진입 구조**를 새로 만든다.

핵심은 아래 한 줄이다.

> 쇼핑몰을 모방하지 않는다.
> 필요한 셸만 가져오고, 그 아래에 우리가 만든 예약 시스템을 붙인다.

이 문서는 새 랜딩 페이지(`/landing`)를 먼저 만들고,
검증 후 기존 `/` 교체 여부를 판단하기 위한 구현 기준 문서다.

---

## 핵심 결정

### 1. 기존 메인을 바로 갈아엎지 않는다
- 먼저 새 페이지 `/landing` 을 만든다.
- 기존 `/` 메인과 예약 흐름은 유지한다.
- 새 페이지에서 구조/브랜딩/예약 연결을 검증한 뒤 교체 여부를 결정한다.

### 2. `00rentcar.com` 에서 가져올 것은 셸뿐이다
가져올 범위:
- 최상단 공지/운영 안내 바
- 브랜드 헤더 톤
- 메인 비주얼/썸네일 흐름
- 연락/운영정보 블록 감성

가져오지 않는 범위:
- 쇼핑몰 카테고리
- 회원/주문/최근본상품
- 게시판/커뮤니티
- 상품 상세 진입 구조
- 장바구니/WISH/ADD/OPTION 등 전자상거래 기능

### 3. 새 랜딩의 역할은 “브랜딩 + 바로 예약 진입” 이다
- 둘러보는 쇼핑몰이 아니다.
- 예약 전 신뢰를 형성하는 홈페이지 셸이다.
- 사용자 행동 목표는 **바로 검색/예약 시작**이다.

### 4. 예약 시스템은 랜딩 아래에 바로 붙인다
- 새 랜딩 상단은 브랜딩 레이어
- 그 아래는 기존 예약 시스템 진입 레이어
- 두 레이어를 시각적으로 자연스럽게 연결한다.

## 00rentcar.com에서 실제 차용 범위

## A. 차용 대상 1 — Top Notice Bar
`00rentcar.com` 상단의 운영 안내 배너 톤만 참고한다.

### 유지할 속성
- 얇은 높이의 공지 바
- 전화 / 카카오톡 / 운영가능지역 같은 실무 정보 노출
- 문장형 안내

### 가져올 목적
- 첫 화면에서 바로 운영감/현업감 전달
- “실제 운영 중인 회사” 인상 강화

### 구현 원칙
- 클릭 유도보다 정보 전달 중심
- 텍스트 1줄 또는 2줄 이내
- 과한 애니메이션/슬라이드 금지

---

## B. 차용 대상 2 — Header Tone
`00rentcar.com` 상단 로고/헤더의 “상업 사이트 같은 안정감”만 차용한다.

### 유지할 속성
- 로고 중심성
- 상단 우측에 핵심 접근 메뉴
- 홈/브랜드 신뢰감 중심 헤더

### 우리 쪽 변환
기존 Cafe24 util menu는 제거하고 아래만 남긴다.
- 예약내역
- FAQ
- 문의 또는 전화번호

### 구현 원칙
- 회원가입/로그인/주문조회 같은 전자상거래 문법 배제
- 단순하고 신뢰감 있는 회사 사이트 상단으로 재구성

---

## C. 차용 대상 3 — Hero / Vehicle Showcase
`00rentcar.com` 메인에 있는 상품 노출 감성을,
우리 쪽에서는 **대표 차량 비주얼 영역**으로 번역한다.

### 유지할 속성
- 차량 이미지가 먼저 보이는 구조
- 시각적으로 “차를 운영하는 회사” 인상 전달
- 썸네일/슬라이드/카드 반복 중 하나의 흐름

### 버릴 속성
- 판매가 중심 문법
- 상품코드/옵션/위시리스트
- 쇼핑몰 상품 카드 인터랙션

### 우리 쪽 목표
Hero 영역은 아래를 수행해야 한다.
1. 대표 차량 이미지 노출
2. 서비스 제공 범위 요약
3. 사용자에게 예약 진입 동기 부여

---

## D. 차용 대상 4 — Contact / Trust Info Block
`00rentcar.com` 하단의 연락처/주소/운영시간 블록 감성을 차용한다.

### 유지할 속성
- 전화번호
- 카카오톡
- 주소
- 운영시간

### 목적

### 구현 원칙
- 장문의 회사 소개문보다 실무 정보 우선
- 카드형 또는 2열/4열 정보 그리드로 단순화

## 새 랜딩 페이지 역할 정의
페이지 경로 가칭:
- `/landing`

## 페이지 역할
1. 브랜드/업체 신뢰감 제공
2. 운영 정보 요약 제공
3. 대표 차량 비주얼 제공
4. 예약 시스템으로 즉시 연결

즉,
`/landing` 은 쇼핑몰 홈이 아니라
**회사형 랜딩 + 예약 앱 진입점**이다.

---

## 페이지 구조 설계

## SECTION 1. TopNoticeBar
### 목적
운영 핵심 정보를 첫 줄에서 전달한다.

### 포함 정보
- 수도권 전지역 배차/반차 가능
- 전화 상담 번호
- 카카오톡 채널 또는 ID
- 운영시간 축약 정보

### UI 원칙
- 높이 낮음
- 한 줄 텍스트 중심
- 좌우 분리 또는 한 줄 결합 가능
- 색상은 브랜드 강조 가능하되 과도한 배너 느낌 금지

### 상태/동작
- 정적 정보 우선
- 클릭 가능한 전화/Kakao 링크는 선택적으로 제공

---

## SECTION 2. BrandHeader
### 목적
브랜드 신뢰감 있는 상단 내비게이션 제공

### 포함 요소
- 로고
- 브랜드명
- 우측 메뉴 2~3개
  - 예약내역
  - FAQ
  - 문의하기 또는 전화번호

### 제외 요소
- 로그인/회원가입
- 주문조회
- 최근본상품
- 고객센터 게시판 드롭다운

### UI 원칙
- 데스크탑: 3단 또는 2단 정렬
- 모바일: 로고 + 햄버거 또는 축약 메뉴
- sticky 여부는 2차 판단, 1차는 비고정 가능

---

## SECTION 3. HeroShowcase
### 목적
사이트 첫 인상과 예약 동기를 만든다.

### 왼쪽(텍스트) 영역
- 메인 카피
  - 예: 서울·수도권 단기렌트 / 딜리버리 가능
- 보조 카피
  - 방문수령 / 왕복 딜리버리 / 보험대차 가능 등
- CTA
  - 아래 예약 블록으로 스크롤
  - 또는 차량 보기

### 오른쪽(비주얼) 영역
- 대표 차량 이미지 1~3장
- 초기 버전은 정적 카드/썸네일 가능
- 이후 필요 시 슬라이드로 확장

### 절대 하지 않을 것
- 상품 가격 리스트 반복
- 상품 상세 페이지 링크 중심 구조
- 장바구니/구매 유도 버튼

### UI 원칙
- Hero 자체가 쇼핑몰 상품 진열판처럼 보이면 안 된다.
- “브랜드/차량/운영범위 소개” 정도로 보이게 해야 한다.

---

## SECTION 4. ReservationEntrySection
### 목적
브랜딩에서 실제 예약 행동으로 전환한다.

### 내용
- 기존 SearchBox 또는 래핑된 예약 진입 컴포넌트
- 메인 예약 시작 문구
- 필요 시 아주 짧은 보조 설명

### 1차 구현 원칙
- 기존 예약 시스템 코어를 최대한 재사용한다.
- 새 랜딩 안에 삽입하는 방식으로 연결한다.
- 메인 로직을 재작성하지 않는다.

### 역할 분리
- 랜딩은 브랜딩/안내
- 예약 엔트리 섹션은 앱 진입

즉 시각적으로는 연결되지만 책임은 분리한다.

---

## SECTION 5. ContactInfoStrip
### 목적
사용자가 예약 전 마지막으로 운영 정보를 쉽게 확인하게 한다.

### 포함 정보
- 전화번호
- 카카오톡
- 주소
- 운영시간
- 방문수령/딜리버리 가능 안내

### UI 형식
추천 2안:
1. 카드형 4칸 그리드
2. 좌측 요약 + 우측 상세 연락처 블록

### 위치
ReservationEntrySection 아래를 기본으로 한다.

### 이유
- 먼저 예약 진입을 보여주고
- 그다음 신뢰 보강 정보를 제공하는 흐름이 자연스럽다.

---

## SECTION 6. Footer
### 목적
기존 Footer는 재사용하되, 필요 시 랜딩 톤에 맞춰 최소 수정한다.

### 유지 요소
- 약관 링크
- 사업자 정보
- 주소/전화 정보

### 검토 포인트
- 랜딩 상단에 연락처가 이미 충분하면 Footer는 과하게 반복하지 않도록 조정 가능

## 라우팅 설계
1차는 아래처럼 별도 운영한다.

```txt
/           -> 기존 메인 유지
/landing    -> 새 랜딩 실험 페이지
/cars       -> 기존 유지
/cars/:id   -> 기존 유지
```

## 이유
- 비교 가능
- 회귀 위험 감소
- 예약 플로우 보호

### 추후 결정
검증 후 아래 중 하나 선택
1. `/` 를 `/landing` 으로 교체
2. `/` 는 랜딩, 예약 앱은 별도 내부 라우트
3. `/landing` 을 운영/테스트 전용으로 유지

## 예약 시스템 연결 방식

## 1차 권장안: 래핑 삽입 방식
새 랜딩 내부에 기존 예약 시작 UI를 넣는다.

### 방식
- `ReservationEntrySection` 안에서 `SearchBox` 재사용
- 검색 결과 진입은 기존 `/cars` 로 유지

### 장점
- 구현 속도 빠름
- 검색 로직 재사용 가능
- 기존 예약 흐름 안정성 유지

### 주의
- SearchBox가 메인 랜딩과 시각적으로 자연스럽게 붙도록 wrapper 필요

---

## 데이터/상태 원칙
새 랜딩은 가능하면 **정적/표시용 상태**만 가진다.

### 랜딩 전용 상태
- Hero 썸네일 선택 상태 (필요 시)
- 모바일 메뉴 열림 여부
- top notice 노출 여부

### 기존 예약 상태
- 전부 기존 SearchBox / 검색 쿼리 체계 재사용

### 원칙
랜딩이 검색 상태를 직접 많이 들고 있지 않게 한다.
검색은 기존 시스템이 담당한다.

---

## 디자인 원칙

### 1. 홈페이지 셸 + 앱 진입의 이중 구조
상단은 브랜딩 페이지,
중간부터는 예약 앱처럼 보여도 괜찮다.

### 2. 쇼핑몰 냄새 제거
- 가격표 반복
- 카테고리 진열
- 상품 탐색 유도
이런 요소를 없애야 한다.

### 3. CTA는 하나로 모은다
핵심 행동은 `예약 시작` 또는 `차량 예약하기` 로 통일.

### 4. 썸네일은 “증거”이지 “상품 리스트”가 아니다
차량 이미지는 신뢰 형성용으로 쓰고,
상품 리스트 역할은 기존 `/cars` 흐름에서 처리한다.
