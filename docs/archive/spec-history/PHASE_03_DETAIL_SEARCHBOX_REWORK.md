# PHASE 03. DETAIL SEARCHBOX REWORK

## 목적
상세 페이지 상단의 검색 박스를 메인 검색 박스 재사용 구조에서 분리하고,
**예약 조건 확인 + 위치/수령 방식 조정 전용 UI**로 재설계한다.

이 문서는 2026-04-04 기준 상세 상단 영역의 최종 구현 기준이다.

---

## 핵심 결정

### 1. 메인과 상세의 역할을 분리한다
- 메인 SearchBox: 검색 시작점
- 상세 SearchBox: 선택된 차량 기준 예약 조건 확인/조정

겉모양이 비슷해도 책임이 다르므로 컴포넌트를 분리한다.

### 2. 상세에서는 날짜/시간을 수정하지 않는다
- 상세는 이미 리스트 결과에서 선택한 차량 기준 화면이다.
- 여기서 대여/반납 일시를 바꾸면 이 차량이 해당 시간에도 가능한지 다시 리스트 단계에서 검증해야 한다.
- 따라서 상세에서 날짜/시간 수정 UI를 제공하지 않는다.

### 3. 상세에서 수정 가능한 것은 위치/수령 방식만 남긴다
- `pickupOption` (`pickup` / `delivery`)
- `dongId`
- `deliveryAddress` 또는 위치 라벨
- `deliveryAddressDetail`
- `deliveryMemo`

### 4. 날짜 변경 액션은 상세에 두지 않는다
- `일정 변경` 버튼 없음
- `다시 검색` 버튼 없음
- 사용자는 상단/브라우저 뒤로가기 등 기존 내비게이션으로 이전 단계로 복귀한다.

---

## 화면 역할 재정의
상세 상단 박스는 더 이상 검색 폼이 아니다.

상세 상단 박스는 아래 2가지를 동시에 수행한다.
1. 현재 차량이 어떤 조건으로 선택되었는지 보여준다.
2. 예약 진행 직전에 위치/수령 방식을 조정하게 한다.

즉, **검색 재입력 UI가 아니라 예약 컨텍스트 고정 + 운영 조건 조정 UI**다.

## 상태 설계

### A. 고정값 (query 기반 읽기 전용)
상세 진입 시 `location.search` 에서 읽고 이후 수정하지 않는다.

```js
fixedSearchInfo = {
  deliveryDateTime,
  returnDateTime,
  driverAge,
}
```

이 값들은:
- 차량 선택의 전제 조건
- 상세 재조회 기준
- 가격/보험/대여기간 표시 기준
으로 사용한다.

### B. 조정값 (상세 로컬 상태)
상세에서 바뀔 수 있는 값만 별도 로컬 상태로 관리한다.

```js
detailAdjustState = {
  pickupOption,
  dongId,
  deliveryAddress,
  deliveryAddressDetail,
  deliveryMemo,
}
```

### C. source of truth 원칙
- 날짜/시간/연령 = `fixedSearchInfo`
- 수령 방식/위치 = `detailAdjustState`
- 예약자 정보 = `reservationForm`
- 약관 = `termsState`

서로 다른 책임의 상태를 섞지 않는다.

## 데이터 흐름

### 상세 진입
1. 리스트에서 상세 진입
2. URL query 파싱
3. `fixedSearchInfo` 생성
4. `detailAdjustState` 초기화
5. `fixedSearchInfo + carId` 기준으로 상세 조회

### 상세 내 변경
#### `pickupOption` 변경
- `pickup` ↔ `delivery` 전환 가능
- `delivery` 로 바꾸면 위치 조정 UI 노출
- `pickup` 으로 바꾸면 딜리버리 입력 UI 숨김
- 기존 위치값은 메모리 유지

#### 위치 변경
- `dongId`, `deliveryAddress` 갱신
- 가격/딜리버리 가능 여부 재계산
- 차량 리스트 재검색은 하지 않음

### 상세 조회 기준
상세 조회 자체는 여전히:
- `carId`
- `fixedSearchInfo`
를 기준으로 유지한다.

즉, 위치/수령 방식 조정은 예약 운영 조건에만 영향을 주고,
선택 차량의 시간 가용성 판정 자체는 건드리지 않는다.

## UI 설계

## 1층: 고정 정보 요약 영역
데스크탑 기준 3블록 구조.

### 블록 1. 대여 일시
- 라벨: `대여 일시`
- 값: `04.04(토) 13:00` 형식
- 읽기 전용
- 입력창 없음

### 블록 2. 반납 일시
- 라벨: `반납 일시`
- 값: `04.05(일) 13:00` 형식
- 읽기 전용
- 입력창 없음

### 블록 3. 운전자 연령
- 라벨: `운전자 연령`
- 값:
  - `만 21세~25세`
  - `만 26세 이상`
- 읽기 전용

### 시각 원칙
- 입력창처럼 보이지 않게 한다.
- 카드/요약칩/정적 값 형태로 처리한다.
- 사용자가 “여긴 안 바뀌는 조건”이라고 바로 이해해야 한다.

---

## 2층: 조정 가능 영역
데스크탑 기준 2열 구조.

### 왼쪽. 수령 방식
- 라벨: `수령 방식`
- 선택 버튼 2개
  - `직접수령`
  - `왕복 딜리버리`

#### 스타일 원칙
- 메인 상단 토글과 시각 언어는 유사하게 유지 가능
- 단, 상세에서는 조정 UI이므로 더 compact 하게 처리
- active 상태가 명확해야 함

### 오른쪽. 위치
- 라벨: `위치`
- `pickupOption` 에 따라 내용 분기

#### 직접수령일 때
- 업체 주소 표시
- 안내 문구: `업체 방문 수령/반납`
- 위치 수정 버튼 없음 또는 비활성 노출 없음
- 딜리버리 상세주소 입력 숨김

#### 왕복 딜리버리일 때
- 현재 선택 위치 라벨 표시
- `위치 선택` 버튼 노출
- 선택된 위치 없으면 placeholder 문구 노출
- `상세 주소` 입력 노출
- `업체 전달 메모` 입력 노출

## 세부 UI 디테일

### 1. 정보 위계
상단 박스 안에서 중요도는 아래 순서다.
1. 날짜/시간 고정 정보
2. 수령 방식
3. 위치
4. 상세 주소/메모

### 2. 문구 길이
- 짧고 직접적으로 유지
- 설명문은 작은 note 1줄 이내
- 긴 가이드 문구 금지

### 3. 버튼 배치
- `수령 방식` 토글은 좌우 2개 또는 한 줄 2버튼
- `위치 선택` 버튼은 delivery 상태에서만 노출
- 검색 버튼/재검색 버튼은 없음

### 4. 읽기 전용 표현
날짜/시간/연령은 아래 둘 중 하나로 구현한다.
- `field summary-field`
- `compact static-card`

중요한 것은 **입력 가능해 보이지 않게** 만드는 것이다.

### 5. 모바일 기준
모바일에서는 아래 순서로 세로 스택.
1. 대여 일시
2. 반납 일시
3. 운전자 연령
4. 수령 방식
5. 위치
6. 상세 주소
7. 업체 전달 메모

### 6. 시각적 톤
- 메인 SearchBox와 동일 계열 스타일은 유지 가능
- 그러나 상세는 더 압축되고 덜 “검색창”처럼 보여야 함
- 요약 + 조정 박스처럼 보여야 함

## 검증 규칙

### 상세 상단 자체 검증
#### 공통
- 날짜/시간/연령은 표시만 하므로 입력 검증 없음

#### 직접수령
- 추가 위치 입력 검증 없음

#### 왕복 딜리버리
- `selectedDongId` 필수
- `deliveryAddressDetail` 필수

## 가격/사이드바 연동 규칙
사이드바는 아래 기준으로 계산한다.
- 기본 대여료 = 상세 조회 응답 기준
- 보험 = 상세 조회 응답 기준
- 딜리버리 비용 = `pickupOption === 'delivery'` 이고 `dongId` 유효할 때 반영
- 총액 = 위 항목 합산

### 재계산 트리거
- `pickupOption` 변경
- `dongId` 변경

### 재계산 제외
- `deliveryAddressDetail`
- `deliveryMemo`

이 둘은 제출 검증용이지 가격 계산 기준이 아니다.

---

## URL 정책
상세에서 조정값을 바꿔도 URL을 즉시 동기화하지 않는다.

### 이유
- 상세는 검색화면이 아님
- 로컬 입력 중 상태를 URL에 계속 반영하면 복잡도만 증가
- 뒤로가기 의미가 흐려진다

### 원칙
- 초기 진입만 query 기반
- 이후 조정은 local state 기반

## 구현 순서

### STEP 1. 문서/결정 고정

- `99_DECISIONS.md` 반영

### STEP 2. 상세 전용 컴포넌트 생성
- `DetailSearchBox.jsx` 생성

- 기존 `searchState.pickupOption` 직접 참조 제거

- 사이드바가 `detailAdjustState` 기준으로 동작하게 정리

### STEP 6. 스타일 정리
- 상세 전용 class 추가
- 데스크탑/모바일 레이아웃 검증

### STEP 7. 회귀 테스트
- 메인 검색 유지
- 목록 진입 유지

## 완료 기준
- 상세 상단에서 날짜/시간 수정 UI가 완전히 제거된다.
- 방문/딜리버리와 위치 조정이 상단 박스 하나로 정리된다.
- 기존 `차량 대여 방법`, `딜리버리 신청` 섹션이 중복 없이 통합된다.
- 가격/검증 로직이 새 상태 구조와 일치한다.
- 메인 SearchBox는 영향을 받지 않는다.

- 상세에서 재검색 버튼 추가
- 지도 API 연동
- 예약 제출 API 구조 변경

이번 범위는 **상세 예약 조건 박스 분리 + 상태 정리 + UI 통합**까지다.
